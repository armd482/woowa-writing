# 웹 프론트엔드-백엔드 API 버저닝 전략과 배포 독립성 확보 방법

## 1. 배포 독립성: 웹 환경의 근본적 과제

### 1.1 프론트엔드-백엔드 배포가 동시일 수 없는 이유

>"프론트엔드를 배포했는데 백엔드가 준비되지 않아 500 에러가 발생했습니다."
>
>"백엔드를 먼저 배포했더니 구버전 프론트엔드에서 필드를 찾을 수 없다는 에러가 쏟아졌습니다."

웹 애플리케이션을 개발할 때 피할 수 없는 문제가 있습니다. 프론트엔드와 백엔드의 배포는 절대 완벽하게 동시에 이루어질 수 없다는 것입니다.

백엔드 API를 먼저 배포하면, 구버전 프론트엔드 코드가 사용자의 브라우저 캐시에 남아있습니다. 반대로 프론트엔드를 먼저 배포하면, 새로운 API 엔드포인트를 호출할 때 백엔드가 준비되지 않았을 수 있습니다.

상황은 백엔드가 여러 서버 인스턴스로 구성된 경우 더욱 복잡해집니다. 롤링 배포 방식을 사용하면 일부 인스턴스는 신버전, 일부는 구버전으로 동작하는 시간대가 반드시 존재합니다.

여기에 모바일 앱이나 여러 프론트엔드 클라이언트가 존재한다면 상황은 더욱 복잡해집니다. 사용자는 앱 업데이트를 즉시 하지 않으며, 웹 브라우저마다 캐싱 정책이 다릅니다. 결국 다양한 버전의 클라이언트가 동시에 API를 호출하는 환경을 피할 수 없습니다.

바로 이러한 배포 비동시성 문제를 해결하기 위해 Stripe, GitHub, Shopify 같은 기업들은 날짜 기반 버저닝과 병렬 변경 패턴을 정교하게 발전시켰습니다. 이들의 전략은 프론트엔드와 백엔드가 독립적으로 배포되면서도 시스템이 안정적으로 작동할 수 있도록 하는 데 초점을 맞춥니다.

### 1.2 배포 독립성의 네 가지 차원

배포 독립성이란 단순히 "프론트엔드와 백엔드를 따로 배포할 수 있다"는 의미를 넘어섭니다. 실무에서 배포 독립성은 네 가지 차원으로 이해할 수 있습니다.

**시간적 독립성**은 프론트엔드와 백엔드가 각자의 일정에 맞춰 배포할 수 있음을 의미합니다. 긴급한 보안 패치가 필요할 때 백엔드만 즉시 배포하거나, UI 개선이 필요할 때 프론트엔드만 배포할 수 있어야 합니다.

**공간적 독립성**은 여러 프론트엔드 클라이언트가 각기 다른 API 버전을 사용할 수 있음을 뜻합니다. 웹 애플리케이션은 최신 버전을 사용하고, iOS 앱은 아직 구버전을 사용하는 상황이 자연스럽게 관리되어야 합니다.

**방향적 독립성**은 어느 쪽이 먼저 배포되어도 시스템이 정상 작동해야 한다는 원칙입니다. 백엔드가 먼저 배포되어도 구버전 프론트엔드가 문제없이 작동하고, 프론트엔드가 먼저 배포되어도 신기능을 안전하게 처리할 수 있어야 합니다.

**롤백 독립성**은 한쪽에서 문제가 발생해 롤백할 때 다른 쪽에 영향을 주지 않아야 함을 의미합니다. 백엔드를 이전 버전으로 되돌렸을 때 프론트엔드가 여전히 정상 작동해야 하며, 그 반대도 마찬가지입니다.

### 1.3 배포 독립성 확보를 위한 핵심 원칙

Stripe에서는 다년간의 API 운영 경험을 통해 성공적인 API 업그레이드를 위한 세 가지 핵심 원칙을 정립했습니다. 이 원칙들은 배포 독립성을 달성하기 위한 철학적 기반이 됩니다.

**가벼움(Lightweight)** 원칙은 업그레이드가 API 소비자와 제공자 모두에게 최소한의 비용과 노력을 요구해야 한다는 것입니다. 거대하고 파괴적인 마이그레이션 대신, 작고 점진적인 변경을 통해 개발팀이 큰 부담 없이 최신 상태를 유지할 수 있도록 유도합니다.

**일급 개념(First-class)** 원칙은 버전 관리를 부가 기능이 아닌 API의 핵심 기능으로 다루어야 한다는 것입니다. 버전 정보가 시스템에 내재화되면, API 문서, 클라이언트 라이브러리, 변경 로그 같은 개발자 지원 도구를 자동화하고 개인화할 수 있습니다.

**고정 비용(Fixed-cost)** 원칙은 이전 버전을 유지하는 데 드는 추가적인 유지보수 비용을 최소화해야 한다는 것입니다. Stripe는 '버전 변경 모듈'이라는 개념으로 과거 버전 지원 로직을 핵심 비즈니스 코드와 분리합니다. 이를 통해 개발자는 새로운 기능을 만들 때 구버전의 복잡성을 고려할 필요가 없어집니다.

## 2. 버전 관리 철학: `/v2`의 함정과 점진적 진화

### 2.1 Major Versioning이 실패하는 이유

많은 개발팀이 API 버전을 `/v1`, `/v2` 형태로 관리합니다. 시맨틱 버저닝의 MAJOR 버전 개념을 URL에 적용한 것으로, 하위 호환성이 깨지는 변경이 있을 때 메이저 버전을 올립니다. 이 방식은 직관적이고 이해하기 쉽다는 장점이 있습니다.

하지만 실무에서 이 접근법은 심각한 문제를 야기합니다. Stripe는 이를 명확히 지적합니다. 버전 간의 변경 사항이 너무 크고 광범위하면 소비자가 새 버전으로 마이그레이션하는 것이 "처음부터 다시 통합하는 것만큼 고통스러운" 작업이 됩니다.

이는 단순히 불편함의 문제가 아닙니다. 프론트엔드 팀은 몇 주간 신규 기능 개발을 중단하고 마이그레이션 작업에 매달려야 합니다. 광범위한 회귀 테스트가 필요하며, 배포 실패 시 롤백 복잡도가 급증합니다. 무엇보다 이런 고통스러운 경험은 팀이 API 업그레이드를 최대한 미루게 만들고, 결국 기술 부채가 누적됩니다.

Stripe는 실제로 `/v1/` 경로를 사용하지만, `/v2` 도입은 적극적으로 회피합니다. 이는 사용자에게 막대한 영향을 줄 만큼 큰 API 재설계가 필요하지 않다는 그들의 철학을 보여줍니다. API를 "전력망이나 수도 공급"에 비유하며, 전력 회사가 2년마다 전압을 바꿔서는 안 되는 것처럼 API도 가능한 한 안정적이어야 한다고 강조합니다.

### 2.2 날짜 기반 버저닝: Stripe, GitHub, Shopify의 선택

이러한 문제를 해결하기 위해 주요 기술 기업들은 날짜 기반 버저닝으로 전환했습니다. GitHub와 Stripe는 `2022-11-28` 형식을, Shopify는 `2023-04`와 같은 분기별 형식을 사용합니다.

이 방식이 처음에는 반직관적으로 느껴질 수 있습니다. 시맨틱 버저닝이 변경의 내용(MAJOR는 호환성 파괴, MINOR는 기능 추가)을 인코딩하는 반면, 날짜 기반 버저닝은 변경이 적용된 시점만을 인코딩하기 때문입니다.

핵심은 다음과 같습니다. 목표는 호환성이 언제 깨졌는지 신호를 보내는 것이 아니라, 애초에 호환성을 깨는 거대한 변경을 거의 만들지 않는 시스템을 구축하는 것입니다.

GitHub는 새 버전 출시 후 이전 버전을 최소 24개월 지원합니다. Shopify는 각 안정 버전을 최소 12개월 지원하며, 연속된 버전 간 최소 9개월의 중복 기간을 보장합니다. 이는 개발팀에게 충분한 전환 시간을 제공하면서도, 변경의 단위를 작게 유지합니다.

### 2.3 변경의 단위를 작게 유지하는 전략

날짜 기반 버저닝의 진정한 가치는 변경의 심리적, 기술적 단위를 재설정하는 데 있습니다.

`/v1`에서 `/v2`로의 전환은 "모든 변경을 한꺼번에 해야 한다"는 압박을 만듭니다. 2년 동안 누적된 모든 개선 사항, 모든 Breaking Changes를 한 번에 적용해야 한다는 부담입니다.

반면 `2024-03-01`에서 `2024-10-01`로의 전환은 "이 기간 동안 도입된 몇 가지 변경"만을 의미합니다. 각 변경은 독립적으로 평가되고, 필요하다면 단계적으로 채택할 수 있습니다. 이는 거대하고 고통스러운 'v2' 마이그레이션을 작고 관리 가능한 증분 업데이트의 연속으로 전환시킵니다.

점진적 롤링 버전 방식은 API가 지속적으로 발전하면서도 안정성을 유지할 수 있게 합니다. 소비자들은 큰 부담 없이 점진적으로 최신 버전으로 업그레이드할 수 있고, 제공자는 기술 부채 없이 지속적으로 API를 개선할 수 있습니다.

## 3. 버전 지정 방식: 헤더 vs URL

### 3.1 헤더 기반 버저닝 (GitHub/Stripe 방식)

GitHub와 Stripe는 HTTP 헤더를 통해 API 버전을 지정하는 방식을 채택했습니다. GitHub는 `X-GitHub-Api-Version` 헤더를, Stripe는 `Stripe-Version` 헤더를 사용합니다.

이 방식의 첫 번째 장점은 URL을 깨끗하게 유지할 수 있다는 점입니다. `/api/users/123`과 같은 직관적인 엔드포인트가 버전과 무관하게 일관되게 유지됩니다. 이는 API 문서화와 클라이언트 코드를 단순하게 만듭니다.

두 번째 장점은 클라이언트가 요청별로 버전을 유연하게 지정할 수 있다는 점입니다. 같은 애플리케이션 내에서도 일부 요청은 안정적인 구버전을 사용하고, 다른 요청은 새로운 기능이 필요한 경우 신버전을 사용할 수 있습니다.

세 번째로 CDN 캐싱 정책과의 충돌이 없습니다. URL 기반 버저닝은 같은 리소스에 대해 여러 URL이 존재하게 되어 캐시 효율성이 떨어질 수 있지만, 헤더 방식은 이 문제를 회피합니다.

GitHub는 헤더가 지정되지 않은 경우 `2022-11-28` 버전을 기본값으로 설정합니다. 이는 명시적인 버전 지정을 권장하면서도, 기존 통합이 갑자기 깨지지 않도록 보호하는 전략입니다.

### 3.2 URL 기반 버저닝 (Shopify 방식)

Shopify는 URL 경로에 버전을 명시하는 방식을 사용합니다. `/admin/api/2023-04/products`처럼 API 경로에 버전이 포함됩니다.

이 방식의 가장 큰 장점은 버전이 명시적으로 보인다는 점입니다. 개발자가 어떤 버전의 API를 사용하고 있는지 URL만 봐도 즉시 알 수 있습니다. 디버깅할 때나 로그를 분석할 때 이는 큰 도움이 됩니다.

두 번째로 특정 버전을 타겟팅하기가 쉽습니다. 브라우저에서 직접 테스트하거나, curl 명령어로 API를 호출할 때 헤더를 추가하는 번거로움 없이 URL만으로 버전을 지정할 수 있습니다.

세 번째로 API 문서화가 직관적입니다. 각 버전의 엔드포인트가 명확히 구분되어, 문서에서 "이 버전에서는 이 경로를 사용하세요"라고 명시하기 쉽습니다.

Shopify는 분기별로 예측 가능한 릴리스 주기를 설정하고, 활발한 서드파티 앱 생태계를 지원하기 위해 '릴리스 후보(Release Candidate)'와 '불안정(Unstable)' 버전을 제공합니다. 개발자들은 다음 안정 버전이 출시되기 전에 미리 테스트할 수 있습니다.

### 3.3 웹 환경에서의 선택 기준

```
헤더 기반 버저닝 (GitHub/Stripe)
┌─────────────────────────────────────┐
│ GET /api/users/123                  │
│ X-API-Version: 2024-10-01           │
└─────────────────────────────────────┘
        ↓
┌─────────────────────────────────────┐
│ 장점:                                │
│ • URL 일관성 유지                    │
│ • 요청별 버전 유연성                 │
│ • CDN 캐싱 효율성                    │
│ • 중앙화된 버전 관리                 │
└─────────────────────────────────────┘

URL 기반 버저닝 (Shopify)
┌─────────────────────────────────────┐
│ GET /admin/api/2023-04/products     │
└─────────────────────────────────────┘
        ↓
┌─────────────────────────────────────┐
│ 장점:                                │
│ • 버전 명시성                        │
│ • 디버깅 편의성                      │
│ • 직관적 문서화                      │
│ • 브라우저 테스트 용이               │
└─────────────────────────────────────┘
```

웹 애플리케이션 개발 관점에서 두 방식은 각각의 장단점이 있지만, 헤더 기반 방식이 일반적으로 더 유리합니다.

첫째, 프론트엔드 코드에서 버전 관리가 중앙화됩니다. HTTP 클라이언트 라이브러리의 설정에서 한 번만 헤더를 지정하면, 모든 API 호출에 자동으로 적용됩니다. URL 기반 방식은 모든 엔드포인트 문자열에 버전을 포함해야 하므로 실수할 여지가 더 많습니다.

둘째, 라우팅 로직이 깔끔하게 유지됩니다. 백엔드에서 버전별로 다른 URL 경로를 관리할 필요 없이, 헤더 값을 읽어 적절한 핸들러로 분기하면 됩니다.

셋째, API 소비자 입장에서 버전 업그레이드가 더 간단합니다. 한 곳의 설정만 변경하면 전체 애플리케이션이 새 버전을 사용하게 되므로, 점진적 마이그레이션을 관리하기 쉽습니다.

다만 URL 기반 방식도 장점이 있습니다. 특히 공개 API를 제공하고 다양한 외부 개발자들이 사용하는 경우, URL만으로 버전이 명확히 보이는 것이 문서화와 지원 측면에서 유리할 수 있습니다.

### 3.4 계정 피닝(Pinning)과 폴 포워드(Fall Forward) 전략

버전 지정 방식과 함께 고려해야 할 것이 기본 동작 정책입니다. 클라이언트가 명시적으로 버전을 지정하지 않았을 때 어떤 버전을 사용할 것인가?

Stripe는 '계정 피닝(Account Pinning)' 전략을 사용합니다. 개발자가 처음 API를 사용하는 시점의 버전에 계정이 고정되고, 명시적으로 업그레이드를 선택하기 전까지는 의도치 않은 변경에 노출되지 않습니다. 이는 금융 거래와 같이 안정성이 최우선인 서비스에 매우 적합한 전략입니다. 개발자는 자신이 준비되었을 때 새 버전으로 전환할 수 있습니다.

반면 Shopify는 '폴 포워드(Fall Forward)' 정책을 사용합니다. 지원이 중단된 버전을 요청하면, 가장 오래 지원되는 안정 버전으로 자동 전환됩니다. 이는 완전한 서비스 실패를 막는 안전장치지만, Shopify 문서는 명확히 경고합니다. 이는 의존해야 할 기능이 아닌 피해야 할 실패 상태라고 말입니다.

폴 포워드가 동작하면 개발자는 자신의 애플리케이션이 어떤 API 버전에서 실행될지 통제권을 잃게 됩니다. 이는 테스트되지 않은 버전으로 강제 전환되는 심각한 리스크를 의미합니다. 따라서 Shopify도 이 동작에 의존하지 말 것을 권고합니다.

웹 애플리케이션 개발 관점에서는 명시적인 버전 지정을 강제하는 것이 가장 안전합니다. 기본값을 제공하되, 프로덕션 환경에서는 반드시 버전을 명시하도록 정책을 수립하는 것이 좋습니다.

## 4. Breaking Changes의 정의와 관리

### 4.1 주요 변경 vs 추가적 변경의 명확한 구분

API 변경이 배포 독립성에 미치는 영향을 평가하려면, 먼저 변경의 성격을 명확히 구분해야 합니다. GitHub는 이를 위한 좋은 기준을 제시합니다.

**주요 변경 사항(Breaking Changes)**은 기존 클라이언트 코드가 수정 없이는 정상 작동하지 않게 만드는 변경입니다. 예를 들어:
- 기존 엔드포인트 제거
- 요청 파라미터나 응답 필드 제거 또는 이름 변경
- 새로운 필수 요청 파라미터 추가
- 인증이나 권한 요구사항 변경
- 파라미터나 응답 필드의 데이터 타입 변경

이러한 변경들은 프론트엔드 코드가 예상하는 API 계약을 깨뜨립니다. 응답에서 특정 필드를 읽으려 했는데 그 필드가 사라졌다면, 프론트엔드는 에러를 발생시키거나 잘못된 동작을 하게 됩니다.

**추가적 변경 사항(Additive Changes)**은 기존 클라이언트에 영향을 주지 않는 변경입니다:
- 새로운 엔드포인트 추가
- 새로운 선택적 요청 파라미터 추가
- 응답에 새로운 필드 추가

기존 클라이언트는 새로운 엔드포인트의 존재를 모르더라도 문제없이 작동합니다. 응답에 새 필드가 추가되어도, 그 필드를 사용하지 않는 구버전 코드는 영향을 받지 않습니다.

### 4.2 Breaking Changes가 배포 독립성에 미치는 영향

Breaking Changes는 배포 독립성을 직접적으로 위협합니다. 백엔드에서 필드를 제거했는데 프론트엔드가 아직 그 필드를 사용하고 있다면, 애플리케이션은 즉시 오작동합니다.

반면 Additive Changes는 이 문제를 일으키지 않습니다. 백엔드가 새 필드를 응답에 추가하고 먼저 배포해도, 구버전 프론트엔드는 그 필드를 무시하고 계속 작동합니다. 나중에 프론트엔드가 배포되면 그때부터 새 필드를 활용하기 시작하면 됩니다.

이것이 점진적 API 진화의 핵심입니다. Breaking Changes는 최소화하고, 대부분의 변경을 Additive Changes로 설계할 수 있다면, 배포 독립성을 자연스럽게 확보할 수 있습니다.

필드 이름을 바꿔야 한다면? 새 이름의 필드를 추가하고(Additive), 구 필드는 유지합니다. 충분한 전환 기간 후 구 필드를 제거하는 것이 Breaking Change입니다. 하지만 이 제거는 모든 클라이언트가 새 필드로 전환했음이 확인된 후에만 이루어집니다.

### 4.3 변경 사항 소통 전략

기술적인 변경만큼이나 중요한 것이 변경 사항을 어떻게 소통하느냐입니다. 명확한 의사소통은 기술적 안정성만큼이나 소비자와의 신뢰를 구축하는 핵심 요소입니다.

**개발자 변경 로그(Changelog)**는 모든 변경 사항을 시간 순서대로 기록하는 문서입니다. 각 변경에 대해 무엇이 바뀌었는지, 왜 바뀌었는지, 그것이 Breaking Change인지 Additive Change인지, 개발자가 취해야 할 조치는 무엇인지를 상세히 설명합니다.

**API 상태 보고서(API Health Report)**는 Shopify가 제공하는 것처럼, 각 개발자의 애플리케이션이 사용 중인 API 중 지원 중단 예정인 리소스가 있는지 분석하여 개인화된 보고서를 제공하는 도구입니다. 개발자는 자신의 애플리케이션이 어떤 영향을 받을지 구체적으로 파악할 수 있습니다.

**상세한 마이그레이션 가이드**는 X(구 트위터)의 사례처럼, 구버전에서 신버전으로 전환하는 과정을 단계별로 안내합니다. 필드 매핑 테이블, 요청 방식의 변화, 구체적인 코드 예시를 포함하여 개발자의 마이그레이션 부담을 최소화합니다.

이러한 소통 도구들은 단순한 문서가 아닙니다. 이들은 API 제공자와 소비자 간의 계약을 명확히 하고, 변경이 예측 가능하게 이루어지고 있음을 보여주며, 개발자들이 충분히 대비할 수 있는 시간을 제공합니다. 투명성과 예측 가능성을 통해 신뢰가 구축됩니다.

## 5. 병렬 변경 패턴: 무중단 마이그레이션의 핵심

### 5.1 3단계 프로세스의 이해

병렬 변경 패턴(Parallel Change Pattern)은 하위 호환성이 없는 특정 인터페이스 변경을 안전하게 수행하기 위한 핵심 전술입니다. 이 패턴은 Expand-and-Contract 패턴으로도 불리며, 명확한 3단계 프로세스로 구성됩니다.

```
병렬 변경 패턴 (Expand-Migrate-Contract)

1단계: 확장 (Expand)
┌──────────────────────────────────────────┐
│ 데이터베이스                              │
│ ┌──────────┐  ┌──────────┐               │
│ │last_name │  │ surname  │ ← 새 컬럼 추가 │
│ │ (기존)   │  │ (NULL OK)│               │
│ └──────────┘  └──────────┘               │
│                                          │
│ 애플리케이션                              │
│ • 이중 쓰기: 두 컬럼 모두에 값 저장        │
│ • 읽기: surname → last_name (폴백)       │
└──────────────────────────────────────────┘
         ↓ 배포 & 검증

2단계: 마이그레이션 (Migrate)
┌──────────────────────────────────────────┐
│ 클라이언트 점진적 전환                     │
│ ┌─────┐  ┌─────┐  ┌─────┐                │
│ │구버전│  │신버전│  │신버전│  ← API 버전 변경│
│ │  5% │  │ 50% │  │ 45% │                │
│ └─────┘  └─────┘  └─────┘                │
│                                          │
│ 모니터링 대시보드                          │
│ • 구버전 API 사용률: 5% → 0%              │
│ • 데이터 마이그레이션 스크립트 실행        │
└──────────────────────────────────────────┘
         ↓ 전환 완료 확인

3단계: 축소 (Contract)
┌──────────────────────────────────────────┐
│ 데이터베이스                              │
│            ┌──────────┐                  │
│            │ surname  │ ← NOT NULL 제약   │
│            │          │                  │
│            └──────────┘                  │
│ ❌ last_name 컬럼 삭제                    │
│                                          │
│ 애플리케이션                              │
│ • surname만 읽고 쓰기                     │
│ • 구버전 코드 제거                        │
└──────────────────────────────────────────┘

✅ 각 단계는 독립적으로 배포 가능
✅ 각 단계마다 안전한 롤백 지점 존재
```

**확장(Expand)** 단계에서는 기존 버전의 인터페이스를 유지하면서 새로운 버전을 추가합니다. 이 단계의 목표는 신구 버전의 애플리케이션이 모두 문제없이 공존할 수 있는 환경을 만드는 것입니다.

**마이그레이션(Migrate)** 단계에서는 모든 클라이언트가 점진적으로 새로운 버전의 인터페이스를 사용하도록 전환합니다. 이 기간 동안 제공자는 모든 클라이언트의 전환 상태를 모니터링해야 합니다.

**축소(Contract)** 단계에서는 모든 클라이언트가 새로운 버전으로의 전환을 완료한 것이 확인되면, 더 이상 사용되지 않는 기존 버전의 인터페이스와 관련 코드를 시스템에서 완전히 제거합니다.

이 3단계 프로세스의 핵심은 위험한 단일 변경을 작고 안전한 여러 단계로 분해한다는 것입니다. 각 단계는 독립적으로 배포 가능하고, 각 단계마다 안전한 롤백 지점이 존재합니다.


### 5.2 확장(Expand): 신구 공존 환경 구축

확장 단계의 핵심 원칙은 기존 버전을 유지하면서 새로운 버전을 추가하는 것입니다. 이는 "아무것도 제거하지 않고 오직 추가만 한다"는 규칙으로 요약됩니다.

데이터베이스 스키마를 예로 들어보겠습니다. `last_name` 컬럼을 `surname`으로 변경해야 한다고 가정합시다. 확장 단계에서는 `surname`이라는 새로운 컬럼을 테이블에 추가합니다. 이때 절대적으로 중요한 원칙이 있습니다. 이 새 컬럼에 `NOT NULL`과 같은 제약 조건을 추가해서는 안 됩니다.

왜일까요? 롤백 시나리오를 생각해봅시다. 새로운 코드를 배포했는데 문제가 발생해 구버전으로 되돌려야 한다면, 구버전 애플리케이션은 `surname` 컬럼의 존재를 모릅니다. 따라서 새 데이터를 생성할 때 이 컬럼에 값을 채워주지 못합니다. 만약 `NOT NULL` 제약이 있다면 모든 쓰기 작업이 실패하게 됩니다.

애플리케이션 레벨에서는 이중 쓰기(Dual-Write)를 구현합니다. 데이터를 생성하거나 수정할 때 `last_name`과 `surname` 두 컬럼 모두에 동일한 값을 씁니다. 읽기 로직은 방어적으로 구현합니다. `surname`을 먼저 읽되, 값이 없으면 `last_name`을 읽는 폴백 로직을 추가합니다.

API 레벨에서도 같은 원칙이 적용됩니다. 응답에 `last_name`과 `surname` 필드를 모두 포함합니다. 구버전 클라이언트는 `last_name`을 사용하고, 신버전 클라이언트는 `surname`을 사용할 수 있습니다. 요청 파라미터도 둘 다 받아들이도록 합니다.

이 단계가 완료되면, 백엔드가 먼저 배포되어도 구버전 프론트엔드가 정상 작동하고, 신버전 프론트엔드가 먼저 배포되어도 신필드를 사용할 수 있는 상태가 됩니다. 배포 순서에 무관하게 시스템이 안정적으로 동작합니다.

### 5.3 마이그레이션(Migrate): 점진적 전환과 모니터링

확장 단계가 완료되어 신구 버전이 공존할 수 있는 환경이 갖춰지면, 이제 클라이언트들을 점진적으로 새 버전으로 전환할 차례입니다.

이 단계의 핵심은 "점진적"이라는 단어에 있습니다. 모든 프론트엔드 애플리케이션을 동시에 업데이트할 필요가 없습니다. 각 팀이나 서비스가 준비되는 대로, 심지어 A/B 테스팅을 통해 일부 사용자만 먼저 새 버전을 사용하도록 할 수 있습니다.

웹 애플리케이션에서는 API 버전 헤더를 변경하는 것으로 전환이 이루어집니다. HTTP 클라이언트 설정에서 `X-API-Version: 2024-10-01`과 같이 새 버전을 지정하면, 해당 애플리케이션의 모든 API 호출이 신버전을 사용하게 됩니다.

이 기간 동안 가장 중요한 것은 모니터링입니다. API 제공자는 각 버전의 트래픽 비율을 추적해야 합니다. 구버전 API의 호출 횟수가 0에 가까워질 때까지 기다려야 합니다. 대시보드를 구축하여 실시간으로 버전별 사용량을 확인하고, 어떤 클라이언트가 아직 구버전을 사용하고 있는지 파악할 수 있어야 합니다.

데이터베이스 레벨에서는 데이터 동기화가 필요합니다. 확장 단계에서는 새로 생성되거나 수정되는 데이터만 새 컬럼에 기록됩니다. 하지만 과거에 생성된 기존 데이터는 여전히 구 컬럼에만 값이 있습니다. 따라서 일회성 데이터 마이그레이션 스크립트를 실행하여 기존 레코드의 `last_name` 값을 `surname` 컬럼으로 복사합니다.

이 데이터 마이그레이션이 완료되면, `surname` 컬럼이 모든 데이터의 최종 정보 소스(source of truth)가 됩니다. 이제 애플리케이션 코드를 수정하여 오직 `surname` 컬럼만 읽고 쓰도록 변경할 수 있습니다. 이 시점부터 코드베이스에서 `last_name`에 대한 모든 참조가 제거됩니다.

### 5.4 축소(Contract): 레거시 제거의 규율

많은 팀이 확장과 마이그레이션 단계는 수행하지만, 레거시 코드를 제거하는 최종 단계를 미루곤 합니다. "일단 돌아가니까", "혹시 모를 롤백을 대비해서"라는 이유로 구버전 코드를 무기한 유지합니다.

하지만 이는 기술 부채를 누적시킵니다. 시스템은 점점 더 복잡해지고, 두 버전을 모두 이해해야 하는 개발자의 인지 부담이 증가하며, 버그가 발생할 표면적이 넓어집니다.

축소 단계는 병렬 변경 패턴의 필수 단계입니다. 모든 클라이언트가 새 버전으로 전환했음이 확인되면, 더 이상 사용되지 않는 레거시 인터페이스를 제거해야 합니다.

데이터베이스에서는 `last_name` 컬럼을 삭제합니다. 이제 더 이상 그 컬럼에 의존하는 코드가 없으므로 안전합니다. 이 시점에서 `surname` 컬럼에 `NOT NULL` 같은 제약 조건을 추가할 수 있습니다.

API에서는 구버전 필드를 응답에서 제거하고, 구버전 파라미터를 받아들이는 로직을 삭제합니다. 버전별 분기 처리 코드도 정리합니다. 코드베이스는 다시 깔끔해집니다.

성공적인 진화는 사용하지 않는 것을 제거하는 규율에서 완성됩니다. 이 규율이 없다면 시스템은 시간이 갈수록 복잡도가 증가합니다.

### 5.5 각 단계에서의 롤백 안전성

병렬 변경 패턴의 가장 큰 장점은 각 단계가 안전한 롤백 지점을 제공한다는 것입니다.

확장 단계 배포 후 문제가 발생하면 어떻게 될까요? 새로 추가된 컬럼이나 필드는 구버전 코드에 영향을 주지 않으므로, 애플리케이션을 이전 버전으로 롤백해도 문제없이 작동합니다. 데이터베이스 마이그레이션도 단순히 컬럼을 추가한 것뿐이므로 되돌리기 쉽습니다.

마이그레이션 단계에서는 더 강력한 안전성이 확보됩니다. 신버전 애플리케이션이 배포된 후 문제가 발견되었다고 가정해봅시다. 이 시점에서 데이터베이스는 두 컬럼 모두에 데이터가 동기화되어 있습니다. 따라서 애플리케이션을 구버전으로 롤백해도, 구버전은 `last_name` 컬럼을 읽고 쓸 수 있으므로 신버전에서 생성된 데이터와도 호환됩니다. 서비스 중단 없이 안정 상태로 복귀할 수 있습니다.

축소 단계는 모든 클라이언트가 새 버전으로 전환했음이 확인된 후에만 진행되므로, 이 단계에 도달했다는 것 자체가 마이그레이션이 성공적으로 완료되었음을 의미합니다. 롤백이 필요한 상황이 발생할 가능성이 극히 낮습니다.

이러한 단계적 접근은 위험을 시간적으로 분산시킵니다. 한 번에 모든 것을 바꾸는 대신, 각 단계를 충분히 검증하고 안정화한 후 다음 단계로 넘어갑니다. 각 단계는 이전 단계의 안정성 위에 쌓아 올려지므로, 전체적인 마이그레이션 위험이 크게 감소합니다.

## 6. 데이터 일관성과 이중 쓰기 문제

### 6.1 병렬 변경에서 발생하는 이중 쓰기 위험

병렬 변경 패턴의 확장 단계에서 이중 쓰기를 구현할 때, 미묘하지만 심각한 문제가 발생할 수 있습니다. 데이터베이스에 상태를 저장하는 작업과 메시지 큐에 이벤트를 발행하는 작업이 모두 성공해야 하는데, 이 두 작업의 원자성을 보장할 수 없다는 것입니다.

구체적인 시나리오를 생각해봅시다. 사용자 정보를 업데이트하는 API가 있습니다. 이 API는 데이터베이스의 `PERSON` 테이블을 업데이트하고, 동시에 Kafka에 `user.updated` 이벤트를 발행하여 다른 마이크로서비스들에게 변경을 알립니다.

확장 단계에서 우리는 `last_name`과 `surname` 두 컬럼 모두에 쓰기를 수행합니다. 그런데 데이터베이스 업데이트는 성공했지만, 네트워크 오류로 Kafka에 이벤트를 발행하는 것이 실패한다면 어떻게 될까요?

데이터베이스에는 업데이트된 정보가 저장되었지만, 다른 서비스들은 이 변경을 인지하지 못합니다. 검색 인덱스를 담당하는 서비스는 업데이트를 반영하지 못하고, 알림을 담당하는 서비스는 알림을 보내지 못합니다. 시스템 전체가 불일치 상태에 빠집니다.

반대 순서로 시도한다고 해서 문제가 해결되지 않습니다. Kafka에 이벤트를 먼저 발행하고 데이터베이스를 업데이트하는 순서로 바꾼다면? 이벤트 발행은 성공했는데 데이터베이스 업데이트가 실패하는 경우, 다른 서비스들은 실제로 발생하지 않은 변경에 대해 처리를 시작합니다.

이것이 바로 이중 쓰기 문제입니다. 두 개의 서로 다른 시스템에 대한 쓰기 작업을 원자적으로 수행할 수 없다는 근본적인 한계에서 비롯됩니다.

### 6.2 트랜잭셔널 아웃박스 패턴의 필요성

이 문제를 해결하기 위한 검증된 패턴이 트랜잭셔널 아웃박스(Transactional Outbox)입니다. 이 패턴의 핵심 아이디어는 간단합니다. 두 개의 서로 다른 시스템에 쓰기를 수행하는 대신, 하나의 데이터베이스 트랜잭션 안에서 모든 것을 처리하는 것입니다.

동작 방식은 다음과 같습니다. 데이터베이스에 'outbox'라는 별도의 테이블을 만듭니다. 이 테이블은 발행해야 할 이벤트들을 저장합니다. 애플리케이션이 사용자 정보를 업데이트할 때, `PERSON` 테이블의 업데이트와 함께 `outbox` 테이블에 이벤트를 기록하는 작업을 하나의 데이터베이스 트랜잭션으로 묶습니다.

이 트랜잭션이 성공하면, 데이터베이스의 상태 변경과 발행해야 할 이벤트가 모두 안전하게 저장된 것이 보장됩니다. 트랜잭션이 실패하면, 두 작업 모두 롤백됩니다. 원자성이 보장됩니다.

그렇다면 이벤트는 언제 Kafka로 전달될까요? 별도의 독립적인 프로세스가 `outbox` 테이블을 주기적으로 폴링하거나, 데이터베이스의 변경 데이터 캡처(CDC, Change Data Capture) 기능을 사용하여 변경 사항을 감지합니다. 이 프로세스가 `outbox` 테이블에서 이벤트를 읽어 Kafka로 전달합니다.

만약 Kafka로의 전달이 실패하면 어떻게 될까요? 이벤트는 여전히 `outbox` 테이블에 남아있으므로, 재시도할 수 있습니다. 네트워크가 복구되거나 Kafka가 다시 사용 가능해지면, 릴레이 프로세스는 성공할 때까지 계속 재시도합니다. 결국 모든 이벤트가 전달됩니다.

### 6.3 원자성 보장 전략

트랜잭셔널 아웃박스 패턴은 두 번의 불안정한 쓰기 작업을, 하나의 원자적 쓰기와 반드시 성공하는 하나의 전달 과정으로 분리합니다.

첫 번째 단계인 데이터베이스 트랜잭션은 ACID 속성을 통해 원자성이 보장됩니다. 상태 변경과 이벤트 기록이 함께 성공하거나 함께 실패합니다. 이는 데이터베이스의 기본 기능이므로 우리가 추가로 해야 할 일이 없습니다.

두 번째 단계인 이벤트 전달은 재시도를 통해 최종적 성공이 보장됩니다. 실패하더라도 이벤트가 소실되지 않고 `outbox` 테이블에 남아있으므로, 시스템이 복구되면 자동으로 전달됩니다.

이 패턴은 약간의 지연을 감수합니다. 이벤트가 즉시 Kafka로 전달되는 것이 아니라, 릴레이 프로세스의 폴링 주기만큼 지연됩니다. 하지만 이는 대부분의 경우 허용 가능한 트레이드오프입니다. 몇 초의 지연을 감수하는 대신, 데이터 일관성을 보장받습니다.

병렬 변경 패턴과 트랜잭셔널 아웃박스 패턴을 함께 사용하면, 무중단 마이그레이션과 데이터 일관성을 동시에 달성할 수 있습니다. `last_name`과 `surname` 두 컬럼에 쓰기를 수행하고, 동시에 `user.updated` 이벤트를 `outbox` 테이블에 기록하는 모든 작업을 하나의 트랜잭션으로 처리합니다. 시스템의 안정성이 극대화됩니다.

## 7. 클라이언트 중심 API 설계: 페이로드 진화 전략

### 7.1 서버 독백에서 클라이언트 협상으로

전통적인 API 설계는 서버 중심적이었습니다. 서버가 제공할 수 있는 모든 정보를 기본 응답에 포함시켰습니다. 클라이언트가 실제로 무엇을 필요로 하는지와 무관하게, 서버가 "이것이 이 리소스에 대한 완전한 표현이다"라고 정의한 모든 데이터를 반환했습니다.

X(구 트위터) API v1.1은 이런 접근의 전형적인 예입니다. 특정 게시물을 조회하면, 게시물의 ID와 텍스트는 물론이고 작성자의 상세 프로필 정보, 위치 정보, 첨부된 미디어의 모든 메타데이터, 통계 정보 등 방대한 데이터를 하나의 응답에 담아 보냈습니다.

이 방식의 문제는 명확합니다. 모바일 네트워크에서 단순히 게시물의 텍스트만 표시하려는 클라이언트도 수십 킬로바이트의 불필요한 데이터를 받아야 했습니다. 네트워크 비용이 증가하고, 응답 시간이 길어지며, 클라이언트는 필요 없는 데이터를 파싱하고 무시하는 로직을 작성해야 했습니다.

X API v2에서는 완전히 다른 철학이 적용되었습니다. 기본 응답에는 게시물의 ID와 텍스트 같은 최소한의 정보만 포함됩니다. 클라이언트가 추가 데이터를 원한다면, 명시적으로 요청해야 합니다.

이는 API 통신 방식이 서버의 일방적인 독백에서 클라이언트와의 협상으로 진화했음을 의미합니다. 서버는 "내가 줄 수 있는 것은 이것이다"라고 말하지 않습니다. 대신 "당신이 필요한 것이 무엇인가?"라고 묻습니다.

### 7.2 X API v2의 필드 요청 방식

X API v2는 클라이언트가 원하는 데이터 구조를 직접 정의할 수 있도록 두 가지 메커니즘을 제공합니다.

**fields 파라미터**는 기본 리소스에서 어떤 필드를 포함할지 지정합니다. 게시물을 조회할 때 `fields=id,text,created_at`를 지정하면, 응답에는 ID, 텍스트, 생성 시간만 포함됩니다. 작성자 정보나 통계 정보 같은 다른 필드들은 포함되지 않습니다.

**expansions 파라미터**는 연관된 객체를 포함할지 지정합니다. `expansions=author_id`를 지정하면, 게시물 작성자의 정보가 별도의 `includes` 객체에 포함되어 반환됩니다. 이 확장된 객체들은 최상위 `data` 객체와 분리되어 있어, 클라이언트가 원하는 방식으로 조합할 수 있습니다.

이 설계의 장점은 여러 가지입니다. 첫째, 네트워크 효율성이 크게 향상됩니다. 클라이언트는 실제로 사용할 데이터만 전송받으므로, 대역폭 사용량과 응답 시간이 줄어듭니다.

둘째, 클라이언트 로직이 단순해집니다. 불필요한 데이터를 파싱하고 무시하는 코드를 작성할 필요가 없습니다. 응답에는 요청한 데이터만 포함되어 있으므로, 처리 로직이 명확해집니다.

셋째, API 진화가 유연해집니다. 서버가 새로운 필드를 추가해도, 기존 클라이언트는 그 필드를 요청하지 않는 한 영향을 받지 않습니다. 이는 Additive Change의 완벽한 구현입니다.

### 7.3 점진적 기능 채택을 가능하게 하는 설계

클라이언트 중심 API 설계는 배포 독립성과 직접적으로 연결됩니다. 백엔드가 새로운 필드나 확장 옵션을 추가할 때, 이는 기존 클라이언트에 아무런 영향을 주지 않습니다. 기존 클라이언트는 계속해서 이전과 동일한 필드만 요청하고, 동일한 응답을 받습니다.

새로운 기능이 필요한 클라이언트만 선택적으로 새 필드를 요청하기 시작합니다. 이는 점진적 채택(Progressive Enhancement)의 원칙입니다. 일부 클라이언트가 새 기능을 사용하기 시작했다고 해서, 다른 클라이언트가 영향을 받지 않습니다.

이 접근 방식은 GraphQL의 핵심 철학과 맞닿아 있습니다. GraphQL에서 클라이언트는 쿼리를 통해 정확히 필요한 데이터 구조를 정의합니다. REST API에서도 fields와 expansions 파라미터를 통해 유사한 유연성을 제공할 수 있습니다.

더 나아가 이는 모바일 우선(Mobile-First) 시대의 요구사항에 부응합니다. 모바일 디바이스는 제한된 네트워크 대역폭과 배터리로 동작합니다. 불필요한 데이터 전송은 사용자 경험을 직접적으로 저해합니다. 클라이언트가 필요한 최소한의 데이터만 요청할 수 있다면, 모바일 앱의 성능이 크게 향상됩니다.

이러한 설계는 API 버전 관리와도 시너지를 냅니다. 큰 변경을 한 번에 적용하는 대신, 새로운 필드를 하나씩 추가하고 클라이언트가 준비되는 대로 채택하도록 유도할 수 있습니다. 각 클라이언트는 자신의 일정에 맞춰 점진적으로 새로운 API 기능을 활용하기 시작합니다. 변화가 작고 유연하게 관리됩니다.

## 8. 실무 적용: 통합된 전략

### 8.1 Stripe의 세 가지 핵심 원칙 재조명

지금까지 살펴본 다양한 전략들은 결국 Stripe가 제시한 세 가지 핵심 원칙으로 수렴합니다. 이 원칙들을 실무 맥락에서 다시 조명해보겠습니다.

**가벼움(Lightweight)** 원칙은 날짜 기반 버저닝과 병렬 변경 패턴을 통해 구현됩니다. 거대한 v2 마이그레이션 대신 작은 변경의 연속으로 API를 진화시킵니다. 각 변경은 독립적으로 평가되고, 클라이언트는 준비되는 대로 채택할 수 있습니다. 프론트엔드 팀은 몇 주간 개발을 중단하고 마이그레이션에 매달릴 필요가 없습니다. 대신 일상적인 개발 흐름 안에서 점진적으로 새 버전으로 전환합니다.

**일급 개념(First-class)** 원칙은 버전 정보를 시스템의 모든 계층에 내재화하는 것을 의미합니다. HTTP 헤더를 통한 버전 지정, 버전별 응답 변환 로직, 버전별 API 문서 자동 생성, 버전별 트래픽 모니터링 대시보드 등 버전 관리가 부차적인 기능이 아닌 시스템의 핵심 설계 요소가 됩니다. 이를 통해 정확하고 일관된 개발자 경험을 제공할 수 있습니다.

**고정 비용(Fixed-cost)** 원칙은 레거시 버전 유지 비용을 최소화하는 것입니다. 병렬 변경 패턴의 축소 단계가 이를 보장합니다. 구버전 지원 로직을 핵심 비즈니스 코드와 분리하고, 모든 클라이언트가 전환을 완료하면 과감히 제거합니다. 개발자는 새로운 기능을 만들 때 구버전의 복잡성을 고려할 필요가 없어집니다. 코드베이스는 깔끔하게 유지되고, 신규 개발자의 온보딩이 쉬워지며, 버그 발생 가능성이 줄어듭니다.

### 8.2 버전 관리를 일급 개념으로 다루기

버전 관리를 일급 개념으로 다룬다는 것은 구체적으로 무엇을 의미할까요? 이는 단순히 버전 번호를 어디에 붙일 것인가의 문제를 넘어섭니다.

첫째, 버전 정보가 요청-응답 사이클 전체에 걸쳐 추적됩니다. 클라이언트가 보낸 버전 헤더는 로깅 시스템에 기록되고, 메트릭 수집 시스템에서 집계되며, 에러 발생 시 컨텍스트로 포함됩니다. 문제가 발생했을 때 "어떤 버전의 API를 사용하는 클라이언트에서 에러가 발생했는가?"를 즉시 파악할 수 있어야 합니다.

둘째, 버전별 차이가 명시적으로 문서화됩니다. Stripe처럼 사용자 계정이 고정된 버전에 맞춰 개인화된 API 문서를 제공할 수 있다면 이상적입니다. 개발자는 자신이 사용하는 버전에 해당하는 정확한 문서만 보게 되어, 혼란이 줄어듭니다.

셋째, 버전 전환 경로가 자동화됩니다. "2024-03-01에서 2024-10-01로 업그레이드하려면 무엇을 변경해야 하는가?"에 대한 답이 마이그레이션 가이드에 명확히 제시되어야 합니다. 더 나아가 자동화된 마이그레이션 도구나 린터를 제공할 수 있다면, 개발자의 부담이 크게 줄어듭니다.

넷째, 지원 중단(Deprecation) 프로세스가 투명합니다. 특정 버전이 언제 지원 중단될 예정인지, 얼마나 많은 클라이언트가 아직 그 버전을 사용하고 있는지, 전환을 완료하기 위해 무엇을 해야 하는지가 명확히 소통되어야 합니다. API 응답 헤더에 `X-API-Deprecation-Warning`과 같은 경고를 포함시켜, 개발자가 즉시 알 수 있도록 할 수 있습니다.

### 8.3 지속 가능한 API 진화를 위한 조직 문화

기술적 패턴과 원칙들이 아무리 훌륭해도, 이를 실행하는 조직 문화가 뒷받침되지 않으면 실패합니다. 지속 가능한 API 진화를 위해서는 다음과 같은 문화적 요소들이 필요합니다.

**축소 단계 완료의 규율**은 가장 중요한 문화적 요소입니다. 많은 팀이 "일단 돌아가니까"라는 이유로 레거시 코드 제거를 미룹니다. 하지만 이는 기술 부채를 누적시키는 지름길입니다. 팀은 마이그레이션을 시작할 때 명확한 완료 기준을 설정하고, 축소 단계까지 완료하는 것을 프로젝트의 필수 요소로 삼아야 합니다.

**Breaking Changes에 대한 신중함**도 필요합니다. 모든 변경이 Breaking Change가 되어서는 안 됩니다. 개발자들은 "이것이 정말 하위 호환성을 깨뜨려야 하는가? 다른 방법은 없는가?"를 항상 자문해야 합니다. 많은 경우 Additive Change로 같은 목표를 달성할 수 있습니다. 필드 이름을 바꾸는 대신 새 이름의 필드를 추가하고, 필수 파라미터를 추가하는 대신 기본값을 제공하는 식입니다.

**클라이언트 관점의 공감**은 API 제공자가 가져야 할 태도입니다. "이 변경이 우리 프론트엔드 팀에게 얼마나 큰 부담이 될까? 외부 API 소비자들은 얼마나 많은 코드를 수정해야 할까?" 같은 질문을 던져야 합니다. API는 제공자만의 것이 아니라, 제공자와 소비자 사이의 계약입니다. 일방적인 변경은 신뢰를 깨뜨립니다.

**투명한 소통의 우선순위**도 강조되어야 합니다. 변경 로그 작성, 마이그레이션 가이드 준비, 지원 중단 공지는 기능 개발만큼 중요한 작업입니다. 이를 "추가 작업"이 아닌 "필수 작업"으로 인식해야 합니다. 좋은 문서는 수백 건의 지원 문의를 줄여주고, 명확한 마이그레이션 가이드는 클라이언트의 전환 시간을 절반으로 단축시킵니다.

**장기적 관점의 의사결정**도 중요합니다. 단기적으로는 레거시 코드를 유지하는 것이 쉬워 보일 수 있습니다. 하지만 장기적으로는 시스템을 복잡하게 만들고 유지보수 비용을 증가시킵니다. 조직은 단기적 편의보다 장기적 지속가능성을 우선시하는 의사결정 문화를 만들어야 합니다.

## 9. 결론: 통제된 변화의 기술
소프트웨어 시스템에서 변화는 피할 수 없는 상수입니다. 비즈니스 요구사항은 끊임없이 진화하고, 기술 스택은 발전하며, 사용자 기대는 높아집니다. 웹 프론트엔드와 백엔드 개발자로서 우리는 이러한 변화의 파도 한가운데 서 있습니다.

하지만 이 글을 통해 살펴본 것처럼, 변화를 두려워할 필요는 없습니다. 변화 자체가 문제가 아니라, 통제되지 않은 변화가 문제입니다. 체계적인 전략과 검증된 패턴을 통해, 우리는 변화를 관리 가능하고 예측 가능한 프로세스로 만들 수 있습니다.

**배포 독립성**은 웹 환경에서 API를 진화시키는 모든 전략의 중심에 있습니다. 프론트엔드와 백엔드가 각자의 속도로 발전할 수 있고, 어느 쪽이 먼저 배포되어도 시스템이 안정적으로 작동하며, 롤백이 필요할 때 독립적으로 수행할 수 있는 시스템. 이것이 우리가 추구해야 할 목표입니다.

**날짜 기반 버저닝**은 거대한 v2 마이그레이션의 고통을 작고 관리 가능한 변경의 연속으로 바꿉니다. 변경의 단위를 작게 유지함으로써, 각 변경이 미치는 영향을 제한하고 위험을 분산시킵니다.

**병렬 변경 패턴**은 무중단 마이그레이션을 위한 핵심 전술입니다. 확장-마이그레이션-축소의 3단계 프로세스를 통해, 위험한 단일 변경을 안전한 여러 단계로 분해합니다. 각 단계는 독립적으로 검증 가능하고, 롤백 지점을 제공합니다.

**명확한 Breaking Changes 정의**는 팀 간 소통의 기반입니다. 무엇이 하위 호환성을 깨뜨리는지 명확히 알면, 어떤 변경이 조율이 필요하고 어떤 변경은 독립적으로 진행 가능한지 판단할 수 있습니다.

**트랜잭셔널 아웃박스 패턴**은 데이터 일관성을 보장하면서도 분산 시스템의 장점을 활용할 수 있게 합니다. 이중 쓰기의 함정을 피하고, 원자성을 확보합니다.

**클라이언트 중심 API 설계**는 점진적 기능 채택을 가능하게 합니다. 클라이언트가 필요한 것만 요청하고, 준비되는 대로 새로운 기능을 채택할 수 있는 유연성을 제공합니다.

이 모든 전략들의 공통분모는 **"크고 위험한 변화를 작고 안전하며 예측 가능한 단계로 나누는 것"**입니다. 빅뱅 방식의 일회성 전환 대신, 점진적이고 지속적인 진화를 선택합니다. 통제할 수 없는 혼돈 대신, 관리 가능한 복잡도를 유지합니다.

Stripe가 말했듯이, "전력 회사가 2년마다 전압을 바꿔서는 안 되는 것처럼" API도 안정적이어야 합니다. 하지만 이는 변화를 거부하라는 의미가 아닙니다. 오히려 변화를 신중하게, 점진적으로, 그리고 소비자와의 신뢰를 유지하면서 수행하라는 의미입니다.

처음 API 버저닝과 마이그레이션 프로젝트를 맡게 되면 부담을 느낄 수 있습니다. "실수로 전체 서비스를 중단시키면 어떡하지? 데이터가 손실되면?" 같은 걱정이 앞설 것입니다. 하지만 이 글에서 소개한 원칙과 패턴들을 따른다면, 그 두려움을 자신감으로 바꿀 수 있습니다.

위대한 엔지니어링은 완벽한 초기 설계에서 나오지 않습니다. 변화를 예측하고, 변화에 대응할 수 있는 시스템을 만들고, 변화가 발생했을 때 안전하게 전환할 수 있는 능력에서 나옵니다. 변화를 통제하고 관리하는 방법을 아는 것, 그것이 진정한 엔지니어링 역량입니다.

다음에 API를 변경해야 할 때, 혹은 데이터베이스 스키마를 마이그레이션해야 할 때, 이 글의 원칙들을 떠올려보세요. 병렬 변경의 3단계를 계획하고, 각 단계의 롤백 지점을 확인하며, 모니터링 대시보드를 준비하세요. 그리고 가장 중요한 것은, 축소 단계까지 완료하는 규율을 지키세요.

변화는 더 이상 두려운 대상이 아닙니다. 그것은 우리가 통제하고, 관리하며, 심지어 활용할 수 있는 기회입니다. 지속 가능한 API 진화, 그것은 단순히 기술적 목표를 넘어 우리가 만드는 소프트웨어가 시간의 시험을 견디고 오래도록 가치를 제공할 수 있도록 하는 철학입니다.
