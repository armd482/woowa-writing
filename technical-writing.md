## 배경

### 이달의 독서왕 랭킹 시스템 설계와 구현

팀 프로젝트 봄봄은 뉴스레터 소비 경험에서 발생하는 두 가지 문제를 해결하기 위해 시작했습니다.

1. 구독한 뉴스레터가 광고/개인 메일과 섞여 도착해 원하는 정보를 찾기 어렵다.
2. 구독만 하고 정기적으로 읽는 습관이 유지되지 않는다.

이를 해결하기 위해 봄봄은 뉴스레터를 한곳에 모아 관리하는 허브 서비스이자, 꾸준한 읽기를 돕는 습관 형성 플랫폼을 지향합니다. 동기부여 장치로 ‘이달의 독서왕’ 월간 랭킹을 도입했습니다. 랭킹은 사용자의 월간 읽기 개수를 기준으로 계산되며, 아래와 같은 목적으로 설계했습니다.

1. 상위 10명의 닉네임·월간 읽기 개수를 공개해 성취감을 제공
2. 모든 사용자에게 현재 순위, 월간 읽기 개수, 바로 앞 순위와의 격차를 보여 “조금만 더 읽으면 순위를 올릴 수 있다”는 즉각적인 동기를 제공

**예시 코드**

```java
@Scheduled(cron = "0 */10 * * * *", zone = "Asia/Seoul")
public void tenMinutelyCalculateMemberRank() {
    log.info("이달의 독서왕 순위 업데이트");
    readingService.updateMonthlyRanking();
    log.info("이달의 독서왕 순위 업데이트 완료");
}
```

요청 시점마다 실시간 계산을 수행하면 데이터베이스 부하가 크기 때문에, Spring Scheduler의 @Scheduled로 10분마다 일괄 집계합니다. 순위와 격차 계산은 애플리케이션에서 반복 처리하지 않고, **데이터베이스(윈도우 함수·집계 쿼리)**로 일괄 계산해 네트워크 왕복을 줄이고 일관성을 확보했습니다.
이 구조를 통해 사용자는 **항상 10분 이내의 최신 랭킹**을 확인할 수 있으며, 시스템은 불필요한 연산 부하를 피하면서도 안정적으로 집계를 수행할 수 있습니다.

### 분산 환경이란

분산 환경은 하나의 시스템을 여러 대의 서버로 나누어 운영하는 구조입니다. 단일 서버가 모든 연산과 요청을 처리하는 방식과 달리, 여러 서버가 네트워크를 통해 협력하여 서비스를 제공하기 때문에 트래픽이 늘어나도 서버를 추가해 확장할 수 있고, 일부 서버가 장애를 일으켜도 서비스 전체가 중단되지 않는 장점이 있습니다. 또한 로드 밸런서를 통해 요청을 분산시켜 효율적인 자원 활용이 가능합니다. 다만 여러 서버가 동일한 데이터베이스를 공유할 경우 동시성 제어와 데이터 일관성 관리가 중요한 과제로 떠오르며, 운영 복잡도 역시 단일 서버 환경보다 높아집니다.

봄봄도 이러한 분산 환경을 기반으로 운영되고 있습니다. 전면에는 로드밸런서가 위치하여 클라이언트 요청을 분산 처리하고, 뒤쪽에는 여러 대의 EC2 애플리케이션 인스턴스가 배치되어 있습니다. 모든 인스턴스는 하나의 MySQL 데이터베이스를 공유하며, 이를 통해 서비스는 더 많은 트래픽을 처리할 수 있고 단일 서버 장애에도 안정적으로 운영될 수 있습니다.

## 문제 상황

모니터링에서 새로 추가된 Trace를 살펴보다가 주기적으로 발생하는 **'Deadlock Found...'** 예외를 발견했습니다.

![img_1.png](img_1.png)

Trace 결과 집계 요청의 전체 실행 시간은 약 1.31초였으며, 이 중 965ms 구간에서 실행된 UPDATE 쿼리에서 데드락이 발생한 것이 확인되었습니다. ReadingScheduler.hourlyCalculateMemberRank 작업 실행 후 updateMonthlyRanking 메서드 호출, 이어서 UPDATE monthly_reading 쿼리 수행까지의 흐름이 기록되어 있어, 예외 발생 지점을 명확히 확인할 수 있었습니다.

**상세한 예외 로그**

```
JDBC exception executing SQL [
    이달의 독서왕 쿼리
] [Deadlock found when trying to get lock; try restarting transaction] [n/a]; SQL [n/a]
```

이 로그는 데이터베이스에서 데드락이 발생했음을 보여줍니다.

로그에 포함된 **`Deadlock found when trying to get lock`** 에러 메시지는 **MySQL/InnoDB가 데드락 상태를 감지했을 때 반환하는 오류 문구**로, 해당 시점에 실행 중이던 **트랜잭션이 더 이상 진행되지 못하고 롤백**되었음을 의미합니다. 이어지는 **`try restarting transaction`** 에러 메시지는 동일한 작업을 다시 시도하면 성공할 가능성이 있음을 나타냅니다.

이 로그만으로도 스케줄러 집계 작업이 데드락으로 인해 정상적으로 완료되지 않았음을 확인할 수 있습니다. 하지만, **충돌한 다른 트랜잭션이나 잠금이 걸린 행과 같은 구체적인 정보는 포함되어 있지 않기 때문에, 원인을 확정하기 위해서는 추가적인 자료가 필요**합니다.


## 문제 원인 찾기

### 문제 이유 추측하기

앞서 제시한 로그와 트레이스 데이터를 바탕으로, 저희 팀은 데드락이 발생한 원인에 대해 다음과 같이 추측하였습니다.

첫째, **'이달의 독서왕' 집계 쿼리의 구조적 복잡성**입니다.

이 쿼리는 하나의 테이블을 여러번 조인하여 순위와 읽기 개수 차이를 계산하도록 구현되어 있었습니다. 단일 테이블을 여러 번 조인하는 구조는 트랜잭션의 실행 시간이 길어지고, 잠금이 걸리는 범위가 넓어질 가능성이 높습니다. 특히 InnoDB 스토리지 엔진에서는 인덱스 기반의 Next-Key Lock이 작동하면서, 실제로 갱신하려는 행뿐 아니라 그 인접한 인덱스 범위까지 함께 잠그는 경우가 발생합니다. 이러한 특성으로 인해, 일부 행에 불필요한 잠금이 확장되었고 결과적으로 다른 트랜잭션의 접근을 막으면서 교착 상태가 발생했을 가능성이 있습니다.

또한, 스케줄러가 랭킹 데이터를 갱신하기 위해 UPDATE를 수행하는 시점에, 사용자가 뉴스레터를 읽으며 해당 테이블에 UPDATE 요청을 보냈을 가능성도 있습니다. 두 작업 모두 같은 테이블을 대상으로 하므로, 타이밍이 겹칠 경우 락 충돌이 발생할 가능성이 충분하다고 판단하였습니다.

둘째, **분산 환경에서 다중 인스턴스의 스케줄러가 동시에 실행된 상황**입니다.

여러 대의 EC2 인스턴스에서 애플리케이션 서버가 구동되고 있으며, 이들 인스턴스는 모두 동일한 데이터베이스를 공유합니다. 스케줄러가 10분 주기로 실행되도록 설정되어 있었기 때문에, 각 인스턴스가 거의 동일한 시점에 갱신 작업을 시작했을 가능성이 있습니다. 이러한 상황에서는 여러 인스턴스가 동시에 같은 테이블을 갱신하게 되어, 서로가 잠금을 획득하려다 데드락이 발생할 수 있습니다.

즉, 데드락은 하나의 단일 원인이라기보다는 **쿼리의 복잡성**, **사용자 요청과의 타이밍 충돌**, 그리고 **분산 환경에서의 동시 실행**이 복합적으로 작용한 결과일 가능성이 높다고 추측하였습니다.

### 문제 이유 좁혀가기

앞선 추측을 바탕으로, 실제로 어떤 요인이 데드락 발생에 더 직접적인 영향을 주었는지를 확인하기 위해 트레이스 데이터를 분석하였습니다.

우선 스케줄러**가 10분마다 주기적으로 실행되지만, 데드락은 매 10분마다 발생하지 않았다**는 점에 주목했습니다. 이는 단순히 쿼리의 복잡성이나 Next-Key Lock으로 인한 잠금 확장이 주요 원인은 아닐 가능성을 나타냅니다. 만약 쿼리 자체의 구조적 문제로 인해 데드락이 발생했다면, 스케줄러가 실행될 때마다 같은 패턴으로 문제가 재현되어야 합니다. 하지만 실제로는 간헐적으로만 발생하였기 때문에, 쿼리 구조보다는 **특정 실행 시점의 환경적 요인(인스턴스 간 스케줄러 충돌)**이 더 큰 영향을 미쳤을 것으로 보았습니다.

또한 **데드락 예외가 스케줄러 실행 트레이스에서만 확인되었다**는 점도 중요한 단서였습니다. 만약 스케줄러의 갱신 작업과 사용자의 뉴스레터 읽기 처리(UPDATE)가 동시에 실행되어 충돌했다면, 사용자 요청을 처리하는 메서드에서도 동일한 데드락 예외가 발생해야 합니다. 그러나 실제로는 사용자 요청 구간에서는 예외가 한 번도 발생하지 않았습니다. 따라서 사용자 요청과의 경합 가능성은 낮다고 판단하였습니다.

이러한 점들을 종합했을 때, 데드락의 주요 원인은 **다수의 인스턴스가 동시에 스케줄러 작업을 실행한 상황**일 가능성이 가장 높다고 보았습니다. 다만, 로그와 트레이스만으로는 이를 확정하기 어려웠기 때문에, 저희 팀은 InnoDB의 내부 잠금 상태를 직접 확인하기 위해 `SHOW ENGINE INNODB STATUS` 명령을 사용하여 추가적인 분석을 진행하였습니다.

### 데드락 원인 검증: InnoDB 내부 상태 분석

데드락의 원인을 보다 정확하게 파악하기 위해, MySQL(InnoDB)이 제공하는 진단 명령어인 `SHOW ENGINE INNODB STATUS`를 확인하였습니다. 이 명령어는 InnoDB 스토리지 엔진의 내부 동작 상태를 출력하며, 특히 최근에 발생한 데드락의 상세 정보를 제공합니다.

이를 통해 단순히 “데드락이 발생했다”는 사실을 넘어서, **어떤 트랜잭션이 잠금을 보유하고 있었는지**, **어떤 트랜잭션이 잠금을 기다리고 있었는지**, 그리고 **결국 어떤 트랜잭션이 롤백되었는지**까지 구체적으로 확인할 수 있습니다

단순히 애플리케이션 로그에 기록된 예외 메시지만으로는 데드락이 발생한 이유를 알기 어렵습니다. 데이터베이스 내부의 실제 잠금 상태를 직접 볼 수 있어 데드락의 원인을 추적할 수 있었습니다. 결과적으로 `SHOW ENGINE INNODB STATUS`는 **데드락을 분석하고 원인을 검증하기 위한 가장 직접적이고 신뢰할 수 있는 근거 자료**로 활용되었습니다.

아래는 실제 실행 결과에서 발췌한 로그입니다.

```
------------------------
LATEST DETECTED DEADLOCK
------------------------
*** (1) TRANSACTION:
TRANSACTION 351844299710552, ACTIVE 0 sec fetching rows
10.0.0.115 bombom_admin executing
UPDATE monthly_reading ...
*** (1) HOLDS THE LOCK(S):
... lock mode S ...
*** (1) WAITING FOR THIS LOCK TO BE GRANTED:
... lock mode S waiting ...

*** (2) TRANSACTION:
TRANSACTION 158951, ACTIVE 0 sec starting index read
10.0.20.26 bombom_admin executing
UPDATE monthly_reading ...
*** (2) HOLDS THE LOCK(S):
... lock_mode X locks rec but not gap ...
*** (2) WAITING FOR THIS LOCK TO BE GRANTED:
... lock_mode X ... waiting ...

*** WE ROLL BACK TRANSACTION (1)
```

위 결과를 통해 두 개의 트랜잭션이 동시에 잠금을 보유하고, 추가 잠금을 기다리면서 교착 상태에 빠진 것을 확인할 수 있었습니다.

먼저 **트랜잭션 (1)** 은 `monthly_reading` 테이블의 **여러 행에 공유 잠금(S락)을 보유**하고 있었습니다. 공유 잠금은 다른 트랜잭션도 동일한 데이터를 읽을 수는 있지만, 쓰기(X락)는 허용되지 않는 상태입니다. 이어서 트랜잭션 (1)은 **다른 행에도 추가로 공유 잠금을 설정하려고 대기 중**이었습니다. 즉, 이미 여러 행의 공유 잠금을 가진 상태에서 더 많은 행을 확보하려는 상황이었습니다.

반면 **트랜잭션 (2)** 는 특정 행에 대해 **배타 잠금(X락)을 보유**하고 있었습니다. 이는 해당 행에 대해 다른 트랜잭션이 읽거나 쓰지 못하도록 완전히 막는 잠금입니다. 동시에 트랜잭션 (2)은 **트랜잭션 (1)이 이미 잠근 다른 행에 대해 X락을 요청하며 대기**하고 있었습니다.

결과적으로 **트랜잭션 (1) 은 새로운 공유 잠금을 얻지 못하고, 트랜잭션 (2)은 배타 잠금을 얻지 못한 채** 서로가 가진 락을 기다리게 되면서 데드락이 발생하였습니다. InnoDB는 이를 감지하고, 트랜잭션 (1)을 롤백하여 데드락을 해소했습니다.

또한, 각 트랜잭션의 IP 주소를 보면, 서로 다른 IP 주소를 갖고 있는 것을 확인하였습니다. 이를 통해 **서로 다른 EC2 인스턴스에서 구동된 애플리케이션 서버들이 동일한 스케줄러 작업을 동시에 실행**했다는 사실을 확인할 수 있었습니다.
즉, 데드락은 단일 서버 내부의 동시성 문제가 아니라, **분산 환경에서 여러 인스턴스가 동일한 스케줄러 작업을 동시에 수행한 것이 직접적인 원인**이었습니다.

## 분산 환경에서 스케줄러로 인한 데드락 해결 방안

### 생각해 본 해결 방법

1. **쿼리 최적화 및 범위 축소**

데드락을 줄이기 위해 가장 먼저 고려한 방법은 쿼리의 범위를 최소화하는 것이었습니다.

현재의 UPDATE 쿼리는 전체 테이블을 대상으로 실행되기 때문에, 잠금 범위가 지나치게 넓어집니다. 이를 개선하기 위해 **읽기용 테이블과 통계용 테이블을 분리하여 쿼리를 최적화**하는 방안을 검토했습니다. 이러한 구조적 개선은 불필요한 락 경합을 줄이고, 동시에 실행되는 트랜잭션 간의 충돌 가능성을 낮출 수 있습니다.

다만, 테이블 설계 변경과 쿼리 재작성에는 재설계와 테스트가 필요하므로 빠른 시간 내에 문제를 해결하기는 어렵다고 판단했습니다. 또한, 이 방법은 **모든 EC2 인스턴스에서 스케줄러가 동시에 실행되는 구조적 문제를 직접적으로 해결하지는 못한다**는 한계가 있었습니다.

2. **외부 분산 락 시스템 도입**

Redis 같은 **외부 분산 락 시스템을 도입하는 방안**을 고려했습니다.

이 접근 방식은 여러 인스턴스 중 오직 하나만 스케줄러 작업을 수행하도록 제어할 수 있어, 충돌을 원천적으로 방지할 수 있습니다. 하지만 외부 인프라 구축과 운영이 필요하다는 점에서 **의존성과 복잡도가 불필요하게 증가**하게 되며, 현재 서비스 규모와 운영 환경에 비해 과도한 접근이라 생각했습니다.

3. **설정 파일을 통한 인스턴스 지정**

스케줄러를 실행할 인스턴스를 설정 파일에서 명시적으로 지정하는 방법도 검토했습니다.

이 방식은 코드 수정 없이 간단히 적용할 수 있고, 단기적으로 문제를 완화할 수 있다는 장점이 있습니다. 하지만 지정한 인스턴스에 장애가 발생할 경우 스케줄러가 동작하지 않게 되어, **운영 안정성 측면에서 리스크**가 있었습니다.

4. **ShedLock 적용**

최종적으로 선택한 방법은 **ShedLock**을 사용하는 것이었습니다.

ShedLock은 데이터베이스를 활용해 스케줄러 작업에 **애플리케이션 레벨 락**을 제공하는 라이브러리로, 먼저 락을 획득한 인스턴스만 작업을 수행하고 나머지 인스턴스는 실행을 건너뜁니다. 별도의 인프라를 추가할 필요가 없고, Spring Scheduler와의 통합이 간단하며, 운영 복잡도를 최소화할 수 있다는 점에서 가장 현실적이고 효율적인 대안이었습니다.

`lockAtLeastFor`와 `lockAtMostFor`의 설정이 적절하지 않을 경우 실행 지연이나 중복 실행이 발생할 수 있으므로, 실제 실행 시간을 기준으로 적정 값을 설정하는 것이 중요합니다.

## ShedLock의 원리와 적용

### ShedLock의 원리

분산 환경에서는 여러 인스턴스가 동일한 스케줄러를 동시에 실행할 수 있습니다. 이 경우 각 인스턴스가 같은 데이터베이스의 동일한 테이블을 갱신하면, Lock 충돌이나 데드락이 발생할 가능성이 높습니다.

ShedLock은 이러한 문제를 해결하기 위해 **데이터베이스 기반의 분산 락** 개념을 적용한 라이브러리입니다.

ShedLock의 핵심 원리는 간단합니다. 스케줄러가 실행될 때, 각 인스턴스는 **데이터베이스의 `shedlock` 테이블에 락 정보를 기록하려고 시도**합니다. 이 과정에서 한 인스턴스가 먼저 `lock_until` 값을 현재 시각보다 미래 시점으로 설정하면서 락을 획득합니다. 락을 획득한 인스턴스만 해당 스케줄러를 실행할 수 있습니다. 다른 인스턴스가 동일한 이름의 락을 시도할 경우 이미 `lock_until`이 미래로 설정되어 있음을 감지하고 실행을 **조용히 건너뜁니다.** 즉, 여러 인스턴스가 존재하더라도 동시에 하나의 작업만 실행되도록 보장하는 방식입니다. (`shedlock` 테이블에 대해서는 이후에 설명해드리겠습니다.)

이때 락은 두 가지 조건 중 하나에 따라 해제됩니다.

1. 스케줄러 작업이 정상적으로 종료되어 락이 명시적으로 해제될 때,
2. 설정된 최대 유지 시간(`lockAtMostFor`)이 지나 자동으로 만료될 때입니다.

이 구조는 비정상 종료나 네트워크 장애가 발생하더라도, 설정된 시간이 지나면 락이 자동으로 해제되어 다음 주기 작업이 정상적으로 수행되도록 보장합니다.

또한, lockAtMostFor와 lockAtLeastFor의 "PT3S"와 같은 값은 ISO-8601 Duration 표준 형식을 따릅니다. 반드시 이 형식을 사용해야 하며, "1s"나 "3sec"처럼 단순 문자열을 사용하면 Spring이 Duration 타입으로 변환하지 못해 **DateTimeParseException**이 발생합니다.

예를 들어 **PT3S는 3초**, **PT10M은 10분**, **PT1H는 1시간**을 의미합니다. 여기서 P는 기간(Period), T는 시간(Time)을 구분하며, H(hour), M(minute), S(second)를 조합해 다양한 시간 단위를 표현할 수 있습니다.

이 표준을 사용하면 시스템 전반에서 일관된 시간 단위 관리가 가능하고, 서버 간 환경 설정이 다르더라도 동일한 기준으로 동작함을 보장할 수 있습니다.

### ShedLock 프로젝트에 적용하기

```
implementation("net.javacrumbs.shedlock:shedlock-spring:6.9.2")
implementation("net.javacrumbs.shedlock:shedlock-provider-jdbc-template:6.9.2")
```

적용 과정은 단순했습니다. 먼저 애플리케이션의 빌드 설정에 ShedLock 의존성을 추가하고, 스프링 설정 파일에서 `@EnableSchedulerLock`을 선언해 락 기능을 활성화했습니다.

```sql
CREATE TABLE IF NOT EXISTS shedlock (
    name        VARCHAR(64)  NOT NULL PRIMARY KEY,
    lock_until  DATETIME(3)  NOT NULL,
    locked_at   DATETIME(3)  NOT NULL,
    locked_by   VARCHAR(255) NOT NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

스케줄러 실행 시 락 정보를 저장하기 위해 데이터베이스에 `shedlock` 테이블을 생성했습니다. 이 테이블은 각 스케줄러 작업의 락 이름(`name`), 락 만료 시각(`lock_until`), 락 획득 시각(`locked_at`), 그리고 락을 획득한 인스턴스(`locked_by`)를 저장합니다.

```java
@Configuration
@EnableScheduling
@EnableSchedulerLock(defaultLockAtMostFor = "PT30S")
public class SchedulerConfig {

    @Bean
    public LockProvider lockProvider(DataSource dataSource) {
        return new JdbcTemplateLockProvider(
            JdbcTemplateLockProvider.Configuration.builder()
                .withJdbcTemplate(new JdbcTemplate(dataSource))
                .usingDbTime() 
                .build()
        );
    }
}
```

설정 클래스에서는 `@EnableSchedulerLock`을 통해 전체 락 정책을 활성화하고, `LockProvider`를 등록하여 데이터베이스 기반으로 락을 관리했습니다. 또한, `usingDbTime()` 설정을 통해 서버 간 시스템 시간 차이로 인해 발생할 수 있는 락 만료 불일치를 방지했습니다.

`@EnableSchedulerLock`에 작성한 `defaultLockAtMostFor`는 각 스케줄러에 `lockAtMostFor`를 설정하지 않았을 때 적용되는 기본값입니다.

만약, `@EnableSchedulerLock`와 `@SchedulerLock`에 모두 값을 설정하지 않는다면, ShedLock은 락의 만료 시간을 알 수 없어, 즉시 락이 해제되고, 분산 락이 제대로 동작하지 않게 됩니다. 

따라서 전역 기본값은 `defaultLockAtMostFor`로 지정하고, 스케줄러별로 다른 값을 설정하고 싶다면 `@SchedulerLock`에 `lockAtMostFor`로 설정해주어야 합니다.

```java
@Component
@RequiredArgsConstructor
public class ReadingScheduler {

    private final ReadingService readingService;

    @Scheduled(cron = "0 */10 * * * *", zone = "Asia/Seoul")
    @SchedulerLock(
        name = "ten_minutely_calculate_member_rank",
        lockAtLeastFor = "PT3S",
        lockAtMostFor  = "PT15S"
    )
    public void tenMinutelyCalculateMemberRank() {
        log.info("이달의 독서왕 집계 시작");
        readingService.updateMonthlyRanking();
        log.info("이달의 독서왕 집계 완료");
    }
}
```

이처럼 각 스케줄러 메서드에는 `@SchedulerLock`을 적용하여 Lock 이름과 유지 시간(`lockAtLeastFor`, `lockAtMostFor`)을 설정했습니다.

이를 통해 동일한 스케줄러가 여러 인스턴스에서 동시에 실행되는 문제를 방지하고, 작업 단위별로 락 지속 시간을 유연하게 제어할 수 있습니다.

쿼리 최적화 이전의 ‘이달의 독서왕’ 집계 스케줄러는 평균 실행 시간이 약 1.3초, 최악의 경우 5초였습니다. 이 실행 시간을 기준으로, `lockAtLeastFor`와 `lockAtMostFor`를 각각 안정성과 효율성 중심으로 조정했습니다.

`lockAtLeastFor`는 **평균 실행 시간보다 약간 긴 값(보통 1~2초 정도의 여유)** 으로 설정하여 너무 빠른 락 해제로 인한 중복 실행을 방지하고, `lockAtMostFor`는 **최악 실행 시간의 약 2~3배**로 설정해 비정상 종료나 지연 상황에서도 락이 안정적으로 해제되도록 했습니다.

이에 따라, `lockAtLeastFor`는 3초(`PT3S`), `lockAtMostFor`는 15초(`PT15S`)로 설정했습니다.

이 설정을 통해 빠른 실행 시에도 일정 시간 동안 락을 유지하여 중복 실행을 방지하고, 비정상 종료 시에도 일정 시간이 지나면 락이 자동 해제되도록 할 수 있었습니다. 실제 운영 환경에서 데드락이 재발하지 않았으며, 스케줄러의 단일 실행이 안정적으로 보장되었습니다.

## 결론

이번 문제는 단순한 쿼리 오류가 아니라, **분산 환경에서 스케줄러가 동시에 실행되며 발생한 데드락**이었습니다. 처음에는 쿼리 구조나 Next-Key Lock 때문이라고 생각했지만, InnoDB의 내부 상태를 직접 확인하면서 원인이 분명해졌습니다.

서로 다른 EC2 인스턴스가 같은 시점에 동일한 스케줄러를 실행했고, 그 결과 한쪽은 읽기 잠금(S락), 다른 한쪽은 쓰기 잠금(X락)을 기다리며 교착 상태에 빠졌던 것입니다. 여러 가지 해결 방법을 검토했지만, 가장 현실적이고 빠르게 적용할 수 있는 방법은 **ShedLock**이었습니다.

ShedLock은 복잡한 인프라나 추가 구성 없이, 데이터베이스를 이용해 “한 번에 하나의 스케줄러만 실행되도록” 보장합니다. 실행이 겹치면 다른 인스턴스는 조용히 스킵하고, 설정된 시간이 지나면 락이 자동으로 풀리기 때문에 운영상 부담이 거의 없습니다.

실제 적용 과정은 단순했지만 효과는 확실했습니다. `lockAtLeastFor`와 `lockAtMostFor`를 실측 실행 시간(평균 1.3초, 최악 5초)에 맞게 설정한 뒤, 운영 환경에서는 더 이상 데드락이 재발하지 않았습니다.

여러 인스턴스가 동시에 배포되어 있어도, 항상 한 번에 하나의 인스턴스만 안전하게 집계를 수행하도록 안정화되었습니다.

이 경험을 통해 **분산 환경에서 스케줄러 제어는 기능 구현보다 실행 동기화 제어가 핵심**임을 배웠습니다.

또한, 단순한 로그만으로는 원인을 찾기 어렵기 때문에, **데이터베이스 엔진 수준에서 상태를 직접 확인하는 과정**이 문제 해결의 전환점이 되었습니다.

ShedLock 도입은 **작은 코드 변경으로 큰 안정성을 확보한 선택**이었습니다. 분산 환경에서도 스케줄러가 한 번만 실행되도록 제어하면서, 서비스의 일관성과 신뢰성을 모두 확보할 수 있었습니다.

이제 ‘이달의 독서왕’ 집계는 10분마다 안정적으로 실행되고, 사용자는 언제나 최신 랭킹을 확인할 수 있게 되었습니다.
