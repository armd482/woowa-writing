# IP와 포트만 알면 상대방에게 데이터를 보낼 수 있을까?

## 들어가며
네트워크를 공부하다 보면 IP 주소와 포트 번호로 통신한다고 배운다. 웹 브라우저에 192.168.0.10:8080을 입력하면 해당 서버와 통신이 
이루어지고, curl 명령어로 특정 IP의 API를 호출할 수 있다. 

그렇다면 IP 주소와 포트 번호만 알면 정말 상대방에게 데이터를 보낼 수 있을까?
결론부터 말하자면 그렇지 않다. 같은 네트워크 내에서 통신하기 위해서는 IP 주소 외에 MAC 주소라는 것이 추가로 필요하다. 

그리고 IP 주소를 통해 MAC 주소를 알아내는 과정을 담당하는 프로토콜이 바로 ARP(Address Resolution Protocol)이다. 
이번 글에서는 왜 MAC 주소가 필요한지, 그리고 ARP 프로토콜이 어떻게 동작하는지 알아보고자 한다.

## IP 주소만으로는 부족하다
OSI 7계층 모델에서 IP 주소는 네트워크 계층(Layer 3)에서 사용되는 논리적 주소다. 논리적 주소라는 것은 실제 물리적인 장치를 
식별하는 주소가 아니라, 네트워크상에서 호스트를 구분하기 위해 부여된 추상적인 주소를 의미한다. 

IP 주소는 DHCP를 통해 동적으로 할당받거나 관리자가 수동으로 변경할 수 있다. 
즉, 같은 컴퓨터라도 재부팅하거나 네트워크 설정을 변경하면 IP 주소가 바뀔 수 있다.

데이터가 전송되는 물리 계층과 데이터링크 계층(Layer 1-2)에서는 IP 주소를 이해할 수 없다. 네트워크 카드와 스위치는 
전기 신호와 프레임을 다루는데, 이들이 이해할 수 있는 주소는 MAC 주소뿐이다. 

스위치는 MAC 주소 테이블을 참조하여 프레임을 적절한 포트로 전달하기 때문이다. 
결국 우리가 알고 있는 IP 주소를 MAC 주소로 변환하는 과정이 필요한데, 이것이 바로 ARP의 역할이다.

## 데이터링크 계층의 역할
네트워크 통신을 제대로 이해하려면 먼저 OSI 7계층 모델의 데이터링크 계층(Layer 2)을 알아야 한다. 우리가 흔히 접하는 IP 주소는 
네트워크 계층(Layer 3)에 속하지만, 실제 물리적인 데이터 전송은 그 아래 계층에서 이루어지기 때문이다.

데이터링크 계층은 같은 네트워크 내에서 직접 연결된 장비 간의 통신을 담당한다. 이 계층의 대표적인 프로토콜이 이더넷(Ethernet)이다.
네트워크 계층이 IP 주소로 통신하는 것과 달리, 데이터링크 계층은 MAC 주소를 사용하여 통신한다.

이 계층에서 동작하는 대표적인 장비가 바로 스위치다. IP 주소는 네트워크상에서 컴퓨터를 식별하는 논리적인 주소다.
하지만 실제로 데이터를 전송할 때는 물리적인 하드웨어, 네트워크 카드에 데이터가 도달해야 한다. 

네트워크 카드는 IP 주소를 이해하지 못한다. 네트워크 카드가 이해할 수 있는 주소는 오직 MAC 주소뿐이다.
같은 네트워크에 여러 대의 컴퓨터가 연결되어 있을 때, 스위치는 각 컴퓨터의 MAC 주소를 보고 데이터를 정확한 네트워크 카드로 전달한다.

IP 주소만 알고 있으면 어떤 컴퓨터인지는 알 수 있지만, 스위치와 네트워크 카드는 MAC 주소가 없으면 실제로 데이터를 전달할 방법이 없다.

## MAC 주소란?
MAC(Media Access Control) 주소는 네트워크 카드 제조 시 하드웨어에 부여되는 48비트 길이의 물리적 주소다. 

AA:BB:CC:DD:EE:FF와 같은 16진수 형태로 표현되며, 이론적으로 전 세계에서 유일한 값을 가진다.
IP 주소가 논리적이고 변경 가능한 주소라면, MAC 주소는 물리적이고 고정된 주소다.

데이터링크 계층에서 동작하는 이더넷 프레임은 출발지와 목적지의 MAC 주소를 헤더에 포함하여 전송된다.
따라서 IP 패킷을 실제로 전송하기 위해서는 목적지의 MAC 주소를 알아야 한다. 

우리는 목적지의 IP 주소는 알지만 MAC 주소는 모르는 상황이다. 이때 필요한 것이 바로 ARP 프로토콜이다.

## ARP 프로토콜의 동작 원리

### 같은 네트워크 내 통신
ARP는 IP 주소를 MAC 주소로 변환하는 프로토콜이다. 예를 들어 192.168.0.5 주소를 가진 컴퓨터 A가 192.168.0.10에 있는
컴퓨터 B와 통신하려고 한다고 가정해보자. 컴퓨터 A는 B의 IP 주소는 알고 있지만 MAC 주소는 모르는 상태다.
이때 컴퓨터 A는 ARP Request를 브로드캐스트한다.

브로드캐스트란 네트워크에 연결된 모든 장비에게 메시지를 전송하는 것을 의미한다.
ARP Request의 목적지 MAC 주소는 FF:FF:FF:FF:FF:FF로 설정되는데, 이는 "모두에게"라는 의미의 특수한 주소다. 

ARP Request 패킷에는 송신자의 IP 주소와 MAC 주소, 그리고 찾고자 하는 대상의 IP 주소가 포함된다.
네트워크의 모든 장비는 이 ARP Request를 수신한다. 하지만 대부분의 장비는 자신의 IP 주소가 아니므로 패킷을 무시한다. 

오직 192.168.0.10 주소를 가진 컴퓨터 B만이 이 요청에 응답한다. 컴퓨터 B는 ARP Reply를 유니캐스트로 전송한다. 유니캐스트는
특정한 한 장비에게만 메시지를 전송하는 것을 말한다. 

ARP Reply에는 자신의 MAC 주소가 포함되어 있으며, 이 패킷은 요청을 보낸 컴퓨터 A에게만 전달된다.
컴퓨터 A는 ARP Reply를 받아 192.168.0.10의 MAC 주소를 알게 되고, 이제 실제 데이터를 전송할 수 있게 된다. 

컴퓨터 A는 ARP Reply를 받아 192.168.0.10의 MAC 주소를 알게 되고, 이제 실제 데이터를 전송할 수 있게 된다. 
목적지 MAC 주소를 알았으니, 이더넷 프레임을 만들어 IP 패킷을 담아서 전송한다.

### 다른 네트워크로의 통신: 라우터의 역할
만약 목적지가 같은 네트워크가 아니라 다른 네트워크에 있다면 어떻게 될까? 예를 들어 192.168.0.5에 있는 컴퓨터가 
8.8.8.8(구글 DNS 서버)과 통신하려고 한다고 가정해보자. 

컴퓨터는 목적지 IP 주소의 앞자리를 비교하여 다른 네트워크인지 판단한다.
내 IP는 192.168.0으로 시작하는데 목적지는 8.8.8이니, 분명 다른 네트워크다.

이 경우 컴퓨터는 8.8.8.8의 MAC 주소를 찾지 않는다. 대신 우리 네트워크의 출입구 역할을 하는 라우터의 MAC 주소를 찾는다. 
라우터는 보통 192.168.0.1과 같은 주소를 가진다. 다른 네트워크로 나가려면 반드시 라우터를 거쳐야 하기 때문이다. 

컴퓨터는 192.168.0.1에 대한 ARP Request를 보내고, 라우터가 자신의 MAC 주소로 응답한다.
이제 데이터를 보낼 준비가 끝났다. 여기서 중요한 점은 목적지 MAC 주소는 라우터의 MAC 주소로 설정하지만, 목적지 IP 주소는 여전히 
8.8.8.8이라는 것이다.

MAC 주소는 "다음으로 전달할 곳"을 가리키고, IP 주소는 "최종 목적지"를 가리킨다. 
라우터가 데이터를 받으면 목적지 IP를 확인하고 다음 라우터에게 전달한다. 이때 다시 목적지 MAC 주소는 다음 라우터의 주소로 바뀌지만,
목적지 IP는 그대로 8.8.8.8을 유지한다. 

이 과정이 여러 라우터를 거쳐 반복되면서 최종 목적지까지 도달한다. 
각 구간마다 MAC 주소는 계속 바뀌지만, IP 주소는 변하지 않는다. 내 컴퓨터는 라우터의 MAC 주소만 알면 되고, 
구글 서버의 MAC 주소는 알 필요가 없다. 나머지는 각 라우터가 알아서 처리한다.

## ARP 캐시
매번 통신할 때마다 ARP Request를 브로드캐스트한다면 네트워크에 불필요한 트래픽이 발생할 것이다.
특히 여러 대의 컴퓨터가 동시에 통신하는 환경에서는 브로드캐스트가 폭증하여 네트워크 성능이 저하될 수 있다. 이러한 문제를 
해결하기 위해 운영체제는 ARP 캐시를 관리한다.

ARP 캐시는 최근에 알아낸 IP 주소와 MAC 주소의 매핑 정보를 일정 시간 동안 메모리에 저장한다. 
캐시 유효 시간 내에 같은 IP로 통신할 경우, ARP Request를 다시 보내지 않고 캐시에 저장된 MAC 주소를 바로 사용한다.

ARP 캐시는 arp -a 명령어로 확인할 수 있다. 캐시 엔트리는 여러 상태를 가질 수 있는데, 
REACHABLE 상태는 최근에 통신이 성공하여 유효한 엔트리임을 의미한다. STALE 상태는 타임아웃이 지나 다음 사용 시
재검증이 필요한 상태이며, DELAY나 PROBE 상태는 현재 재검증이 진행 중임을 나타낸다.

첫 번째 통신 시에는 ARP Request와 Reply를 주고받는 시간이 추가로 소요되지만, 이후 통신에서는 캐시를 활용하여 즉시 데이터를 
전송할 수 있다. 

이는 네트워크 효율성을 크게 향상시키는 중요한 메커니즘이다. 특히 인터넷 통신의 경우 거의 모든 패킷이 라우터로 가기 
때문에, 라우터의 MAC 주소가 캐시에 저장되어 있으면 매번 ARP를 수행할 필요가 없다.

## ARP의 한계
ARP는 1980년대에 설계된 프로토콜로, 당시에는 보안이 크게 고려되지 않았다. 
ARP의 가장 큰 문제는 검증 메커니즘이 없다는 것이다. 

어떤 장비든 ARP Reply를 보낼 수 있고, 받는 쪽에서는 그것이 정당한 응답인지 확인할 방법이 없다.
이러한 특성을 악용한 공격이 ARP 스푸핑(ARP Spoofing)이다. 공격자는 거짓 ARP Reply를 피해자에게 전송하여 ARP 캐시를 
오염시킬 수 있다. 

예를 들어 "라우터의 IP 주소에 대응하는 MAC 주소는 공격자의 MAC 주소다"라는 거짓 정보를 주입하면, 피해자가 
인터넷으로 보내는 모든 패킷이 공격자를 거쳐가게 된다. 이는 중간자 공격(Man-in-the-Middle Attack)의 전형적인 형태다.

현대 네트워크에서는 이러한 문제를 완화하기 위해 Dynamic ARP Inspection(DAI)이나 정적 ARP 엔트리 같은 보안 메커니즘을 사용한다.
하지만 ARP 프로토콜 자체의 근본적인 한계는 여전히 존재한다.

## 마치며
IP 주소와 포트 번호만으로는 실제 데이터를 전송할 수 없다. 
물리적인 네트워크에서 동작하기 위해서는 MAC 주소가 필요하다.

ARP는 브로드캐스트를 통해 네트워크에 질문을 던지고, 해당하는 호스트만 유니캐스트로 응답한다. 
다른 네트워크와 통신할 때는 최종 목적지의 MAC 주소가 아닌 라우터의 MAC 주소를 알아낸다. 
각 네트워크 구간마다 MAC 주소는 바뀌지만 IP 주소는 그대로 유지된다.

ARP 캐시는 매번 브로드캐스트를 보내지 않아도 되게 만들어 네트워크 효율성을 높인다.
하지만 검증 메커니즘이 없어 스푸핑 공격에 취약하다는 한계가 있다.

웹 프로그래밍을 하면서 데이터링크 계층이나 ARP를 직접 다룰 일은 거의 없다. 
하지만 "IP 주소만 알면 통신이 된다"고 생각했던 것이 사실은 그 아래에서 MAC 주소 변환이 일어나고 있었다는 것을 알게 되면, 
네트워크가 어떻게 동작하는지 한층 더 이해할 수 있다.
