## 1️⃣ 클러스터링 인덱스란?

클러스터링 인덱스를 배우기 전에, 먼저 **‘클러스터링(Clustering)’**이라는 단어의 의미부터 짚고 넘어가야 합니다.

사전적으로 클러스터링이란 **‘유사한 특성을 가진 데이터들을 그룹으로 묶는 것’**을 말합니다.

이를 데이터베이스(DB) 관점에서 본다면,

> “유사한 특징을 가진 레코드(record)들을 공통된 컬럼(column) 값으로 묶어 인덱싱한다.”
>
>
> 라고 이해할 수 있습니다.
>

---

### 💭 모든 레코드에 공통된 컬럼이 뭘까?

대부분의 테이블은 **프라이머리 키(Primary Key, PK)** 를 갖고 있습니다.

PK는 각 레코드를 유일하게 식별하는 값이며, **중복되지 않고 반드시 존재해야 하는 식별자**죠.

따라서 InnoDB 엔진은 **Primary Key를 기준으로 데이터를 정렬해서 저장합니다.**

이게 바로 클러스터링 인덱스의 핵심입니다.

---

### 🔑 정리하자면

> InnoDB의 클러스터링 인덱스는
>
>
> **Primary Key를 기준으로 데이터 자체가 인덱스 구조 안에 정렬되어 저장되는 형태**입니다.
>

즉,

- 인덱스와 데이터가 물리적으로 분리되지 않고,
- 데이터가 곧 인덱스인 구조라고 볼 수 있습니다.

---

## 2️⃣ 클러스터링 인덱스의 구조

클러스터링 인덱스는 내부적으로 **B+Tree 구조**로 구성되어 있습니다.

루트 노드(Root Node), 브랜치 노드(Branch Node), 리프 노드(Leaf Node)로 나뉘며,

데이터는 **리프 노드에 실제 값 형태로 저장**됩니다.

```
          [Root Node]
              ↓
         [Branch Node]
              ↓
         [Leaf Node = Data]

```

이 말은 곧,

> “리프 노드에 저장된 인덱스 값 = 실제 데이터 레코드”
>
>
> 라는 의미입니다.
>

그림으로 표현하면 아래와 같습니다.

![스크린샷 2025-10-13 오후 10.36.26.png](attachment:0836abee-c9db-46a0-b7e6-ff7f2b570caf:스크린샷_2025-10-13_오후_10.36.26.png)

즉, 인덱스를 통해 데이터를 찾는 순간 이미 레코드 전체를 읽은 상태가 됩니다.

이는 **MyISAM의 비클러스터링 인덱스 구조**와는 본질적으로 다릅니다.

## 3️⃣ InnoDB의 자동 동작

### ⚙️ 자동 인덱스 생성

걱정할 필요 없습니다.

InnoDB는 테이블 생성 시 **자동으로 PRIMARY KEY 인덱스를 생성**합니다.

```sql
SHOW INDEX FROM employee;

```

위 명령어를 실행하면 `PRIMARY`라는 이름의 인덱스가 자동 생성되어 있음을 확인할 수 있습니다.

---

### 🔍 만약 Primary Key가 없다면?

InnoDB는 다음 순서대로 **클러스터링 인덱스를 대체할 키를 선택**합니다.

1. 명시된 Primary Key가 존재하면 그것을 사용
2. 유일(Unique)하고 NULL이 아닌 인덱스가 존재하면 그것을 사용
3. 둘 다 없으면 **숨겨진 Row ID**를 생성해 클러스터링 인덱스로 사용

즉, InnoDB는 어떤 경우에도 클러스터링 인덱스를 반드시 유지하려고 합니다.

---

## 4️⃣ 클러스터링 인덱스의 장점

클러스터링 인덱스의 장점은 크게 두 가지입니다.

- 검색 성능이 빠르다.
- 락(Lock) 효율이 높다.

---

### 🚀 첫 번째 장점 — 빠른 검색 성능

### 🔹 MyISAM 시절엔?

InnoDB 이전, MyISAM 엔진에서는 인덱스와 데이터가 별도로 저장되었습니다.

따라서 인덱스를 통해 찾은 데이터의 **디스크 주소값**으로 한 번 더 접근해야 했습니다.

```sql
CREATE TABLE employee2 (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL,
  PRIMARY KEY (id)
) ENGINE=MyISAM;

```

```sql
SELECT * FROM employee2 WHERE name = 'pobi';

```

- name 인덱스가 없다면 → **전체 테이블 스캔(Full Table Scan)**
- name 인덱스가 있다면 → **주소를 찾고, 디스크를 한 번 더 탐색**

즉, 두 번의 디스크 접근이 필요했고 **Random I/O**가 과도하게 발생했습니다.
그림으로 표현하면 아래와 같습니다.

### 💡 InnoDB에서는?

InnoDB에서는 PK를 기준으로 **클러스터링 인덱스가 자동 생성**됩니다.

따라서 인덱스 탐색 시 다음과 같은 과정을 거칩니다.

1️⃣ 세컨더리 인덱스에서 PK 값을 조회

2️⃣ PK를 통해 클러스터링 인덱스 리프 노드(=데이터)에 직접 접근

즉, 인덱스가 데이터와 함께 있으므로 **디스크 접근 횟수가 감소**합니다.

| 구분 | MyISAM | InnoDB (클러스터링 인덱스) |
| --- | --- | --- |
| 데이터 저장 | 인덱스와 데이터 분리 | 인덱스에 데이터 포함 |
| 인덱스 반환 값 | 디스크 주소값 | PK 값 |
| 디스크 접근 | Random I/O | Sequential I/O |
| 조회 성능 | 느림 | 빠름 |

---

### 🔒 두 번째 장점 — 락(Lock) 효율성 향상

InnoDB는 트랜잭션과 락 기능을 통해 데이터 정합성을 보장합니다.

하지만 MyISAM은 클러스터링 인덱스가 없어 **테이블 락(Table Lock)** 수준으로만 잠금이 가능했습니다.

InnoDB는 클러스터링 인덱스를 통해 **레코드 단위(Row-Level Lock)** 를 제공합니다.

| 구분 | MyISAM | InnoDB |
| --- | --- | --- |
| 락 범위 | 전체 테이블 | 단일 레코드 |
| 동시성 | 낮음 | 높음 |
| 정합성 | 낮음 | 높음 |

---

## 5️⃣ 클러스터링 인덱스의 단점

장점만 있는 기술은 없습니다. 클러스터링 인덱스에도 주의할 점이 있습니다.

### ⚠️ 1. PK 업데이트 비용이 크다

PK가 데이터 정렬의 기준이기 때문에,

PK를 변경하면 **데이터 전체의 물리적 재배치**가 발생할 수 있습니다.

### ⚠️ 2. 세컨더리 인덱스가 PK를 포함한다

세컨더리 인덱스는 실제 데이터를 찾기 위해 PK 값을 포함합니다.

즉, PK가 길수록 세컨더리 인덱스 크기도 커집니다.

### ⚠️ 3. INSERT 순서가 비효율적일 수 있다

PK가 랜덤한 값(UUID 등)이라면,

새로운 레코드가 중간에 삽입되면서 **페이지 분할(Page Split)**이 자주 발생하게 됩니다.

이로 인해 디스크 I/O 부하가 증가합니다.

---

## 6️⃣ 세컨더리 인덱스와의 관계

InnoDB에서 세컨더리 인덱스(Secondary Index)는

클러스터링 인덱스와 밀접하게 연관되어 있습니다.

- 세컨더리 인덱스의 리프 노드는 **PK 값**을 포함합니다.
- 따라서 세컨더리 인덱스로 검색할 때 항상 두 단계 접근이 필요합니다.

```
[세컨더리 인덱스] → [PK 조회] → [클러스터링 인덱스 데이터]

```

이 과정을 “**더블 루크업(Double Lookup)**”이라고 부릅니다.

---

## 7️⃣ 실제 동작 예시

```sql
CREATE TABLE user (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  email VARCHAR(255),
  nickname VARCHAR(100),
  INDEX idx_email (email)
) ENGINE=InnoDB;

```

다음 쿼리를 실행해봅시다.

```sql
SELECT * FROM user WHERE email = 'test@example.com';

```

- 1단계: `idx_email` 인덱스를 탐색해 해당 email의 PK(`id`)를 찾음
- 2단계: PK로 클러스터링 인덱스에 접근해 전체 데이터 레코드 반환

즉, **email 인덱스 → PK 인덱스 → 데이터** 순으로 검색이 이루어집니다.

---

## 8️⃣ 클러스터링 인덱스 설계 시 주의점

- 가능한 한 **단조 증가하는 값(PK)** 을 사용하세요.

  → 예: AUTO_INCREMENT, 타임스탬프 등

- PK를 자주 변경하지 마세요.

  → 물리적 재정렬로 인해 성능 저하가 발생합니다.

- PK 길이를 최소화하세요.

  → 세컨더리 인덱스 크기에 직접적인 영향을 미칩니다.


---

## 9️⃣ 클러스터링 인덱스 vs 비클러스터링 인덱스

| 항목 | 클러스터링 인덱스 (InnoDB) | 비클러스터링 인덱스 (MyISAM) |
| --- | --- | --- |
| 저장 구조 | 인덱스와 데이터 일체형 | 인덱스와 데이터 분리형 |
| 리프 노드 | 실제 데이터 저장 | 데이터 주소 저장 |
| PK 필요 여부 | 필수 | 선택 |
| 검색 성능 | 빠름 (Sequential I/O) | 느림 (Random I/O) |
| 락 단위 | 레코드 단위 | 테이블 단위 |
| 트랜잭션 | 지원 | 미지원 |

---

## 🔚 결론

클러스터링 인덱스는 단순한 인덱스 기술이 아니라,

**데이터 저장 구조와 성능의 본질을 바꾸는 핵심 메커니즘**입니다.

InnoDB의 클러스터링 인덱스는

- PK 기준으로 데이터를 정렬해 저장하고,
- 빠른 검색과 효율적인 락 처리를 지원하며,
- MySQL의 트랜잭션 안정성과 성능을 크게 개선했습니다.
