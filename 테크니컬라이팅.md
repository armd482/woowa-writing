# 객체지향 설계 원칙으로 유연한 프론트엔드 만들기

### \- 협력, 책임, 역할 중심으로 -

---

## 1\. 들어가며

이 글은 프론트엔드 개발에서 '변경에 유연한 코드'를 작성하기 위해 객체지향 설계 원칙의 핵심인 협력, 책임, 역할을 어떻게 활용할 수 있는지 구체적인 코드 예시와 함께 알아봅니다. 이 글을 통해 다음을 얻을 수 있습니다.

- 데이터가 아닌 책임 중심 설계의 중요성 이해

- 협력, 책임, 역할이란?

- 커스텀 훅을 통해 알아보는 객체지향 설계 원칙

'변경에 대응하기 쉬운 코드'를 작성하기 위한 설계 원칙을 알아보던 중, **객체지향 설계 원칙**이 큰 도움이 된다는 것을 알게 되었습니다. 그런데 여기서 "객체지향 설계 원칙이 객체지향 프로그래밍을 의미하는 거 아닌가?"라는 생각이 들을 수 있습니다. 이 두 가지 개념의 차이를 짚고 넘어가겠습니다.

> #### 💡 객체지향 프로그래밍과 설계 원칙은 다릅니다.
>
> 이 둘의 차이점을 설명하자면 다음과 같습니다.
>
> - **객체지향 프로그래밍(OOP)** 은 코드를 구성하는 **'도구'** 또는 **'문법'** 에 가깝습니다. 🧰
>
>   - 예를 들어, `class` 키워드를 사용해 관련된 데이터와 행동을 하나로 묶어 '객체'라는 부품을 만드는 기술입니다. 이는 글쓰기로 치면 '단어'나 '문법'을 아는 것과 같습니다.
>
> - **객체지향 설계 원칙**은 그 도구들을 사용해 **'어떻게 좋은 구조를 만들 것인가'** 에 대한 **'전략'** 이자 **'철학'** 입니다. 🏛️
>
>   - "이 부품은 어떤 책임을 져야 할까?", "부품끼리는 어떻게 소통해야 나중에 바꾸기 쉬울까?" 같은 질문에 대한 답을 줍니다. 이는 좋은 '문법'을 사용해 '설득력 있고 짜임새 있는 글을 쓰는 방법'에 해당합니다.

객체지향 설계 원칙과 객체지향 프로그래밍은 다릅니다. 프론트엔드에서도 객체지향 설계 원칙을 적용할 수 있는 이유가 여기에 있습니다.

이 글은 class 문법과 같은 객체지향 프로그래밍(OOP)을 다루지 않습니다. 대신 '오브젝트(조영호 님 저)'의 핵심 원칙인 협력, 책임, 역할을 프론트엔드 관점에서 탐구합니다. 이 원칙을 활용해 어떻게 더 유연하고 확장 가능한 컴포넌트와 함수를 만들 수 있는지 알아봅니다. 이 원칙들을 이해하면, 단순히 기능을 구현하는 것을 넘어 **변경에 강한 코드**를 작성하는 단단한 기반을 다지는 데 도움이 되리라 믿습니다.

---

## 2\. 왜 객체지향 원칙인가?

프론트엔드는 "최신 프레임워크와 상태 관리 라이브러리를 잘 쓰는 것만으로도 충분하지 않을까?"라고 생각할 수 있습니다. 물론 훌륭한 도구들은 생산성을 높여주지만, 도구가 설계의 본질적인 문제를 해결해주지는 않습니다. 설계의 본질은 **'변경'을 관리**하는 데 있습니다.

객체지향 설계 원칙을 적용한다는 것은, 코드의 "변경"에 유연하게 대처할 수 있게 해줍니다. 요구사항은 항상 변하기 마련이고, 좋은 설계는 이러한 변화가 시스템 전체를 뒤흔들지 않고 국소적인 영향에 그치도록 만듭니다.

이 글에서 관통하는 가장 중요한 설계 흐름은 다음과 같습니다.

> **협력 → 책임(행동) → 상태**

우리가 만들어야 할 기능, 즉 **'협력'** 의 전체적인 그림을 먼저 그리면, 그 협력에 참여하는 각 객체(컴포넌트)가 수행해야 할 **'책임'** 이 자연스럽게 도출됩니다. 그리고 그 책임을 완수하기 위해 필요한 최소한의 데이터, 즉 **'상태'** 가 결정됩니다.

이 흐름은 '어떤 데이터가 필요할까?'에서 시작하는 데이터 중심적인 접근과 정반대입니다. 데이터가 아닌 책임과 협력을 우선시할 때, 우리는 비로소 컴포넌트의 본질적인 역할에 집중하고 응집도 높고 결합도 낮은 코드를 작성할 수 있게 됩니다.

## 3\. 협력, 책임, 역할이란?

### 3.1. 컴포넌트는 '협력'하고 있나요?

객체지향의 세계에서 객체는 홀로 존재하지 않습니다. 여러 객체가 서로 메시지를 주고받으며 공동의 목표를 달성하는데, 이를 **'협력'** 이라고 합니다. 프론트엔드 애플리케이션도 마찬가지입니다. `Header`, `ProductList`, `ShoppingCart` 컴포넌트는 각자 독립적인 UI 조각이 아니라, '상품을 둘러보고 장바구니에 담는다'는 하나의 큰 협력을 위해 존재하는 파트너들입니다.

이 협력 관계를 어떻게 설계하느냐에 따라 코드의 유연성이 결정됩니다.

#### 잘못된 협력: 데이터 중심 설계

데이터 중심 설계는 **"이 결과를 만들려면 어떤 데이터가 필요하지?"** 라는 질문에서 시작합니다. 우아한 테크 코스에서 진행했던 **'장바구니 미션'** 의 쿠폰 유효성 검사 기능을 예로 들어보겠습니다. 이 미션은 다양한 종류의 할인 쿠폰(고정 금액 할인, 무료 배송 등)을 장바구니에 적용하는 과제였습니다.

**[As-Is: 데이터 중심 설계로 작성된 코드]**

```tsx
export const validateCoupons = ({ cartItems, coupon, today = new Date() }) => {
  // ... 만료일 검증 로직 ...

  const totalPrice = calculateTotalPrice(cartItems);

  switch (coupon.discountType) {
    case CouponType.FIXED: {
      return validateFixedDiscountCoupon({ totalPrice, coupon });
    }
    case CouponType.FREE_SHIPPING: {
      return validateFreeShippingCoupon({ totalPrice, coupon });
    }
    //... 기타 쿠폰 타입들
    default:
      return false;
  }
};
```

당시 이 코드를 작성할 때의 사고 과정은 전형적인 데이터 중심 설계였습니다.

1.  유효성 검사를 위해 뭐가 필요하지?
2.  쿠폰에 만료일이 있으니 `today` 데이터가 필요하겠네.
3.  고정 할인 쿠폰은 `totalPrice` 데이터가 필요하겠네.
4.  ...

이처럼 필요한 데이터를 먼저 생각하고, 이 데이터들을 처리하는 거대한 `validateCoupons` 함수를 만들었습니다. 하지만 이 방식은 여러 문제점을 낳았습니다.

- **높은 결합도와 인지 부하:** validateCoupons 함수를 수정하려면, 개발자는 **모든 쿠폰의 종류와 정책을 알고 있어야 합니다.** 이는 코드를 이해하고 수정하는 데 드는 **인지 부하(Cognitive Load)** 를 직접적으로 높입니다.
- **정보 은닉 위배:** 고정 할인 쿠폰은 `today` 데이터가 필요 없음에도 `validateCoupons` 함수를 통해 `today`의 존재를 알게 됩니다.
- **변경의 어려움:** 만약 고정 할인 쿠폰의 유효성 검사에 새로운 데이터가 필요해진다면, `validateCoupons` 함수의 인자를 수정해야 하고, 이는 이 함수를 사용하는 모든 곳에 영향을 미칩니다.

#### 올바른 협력: 책임 중심 설계

책임 중심 설계는 **"이 결과를 만들려면 어떤 협력과 행동이 필요하지?"** 라는 질문에서 시작합니다.

1.  **협력 정의:** 유효한 쿠폰을 얻는다.
2.  **행동(책임) 정의:** 각 쿠폰이 스스로 유효한지 검증한다.
3.  **책임 할당:** "유효성 검증"이라는 행동을 가장 잘 할 전문가는 **쿠폰 자신**입니다.
4.  **인터페이스 결정:** 모든 쿠폰은 "나는 유효한가?"라는 질문에 답할 수 있어야 합니다. 이 행동의 이름을 `isValid()`라고 정하고, 필요한 외부 정보는 `ValidationContext`라는 형태로 받기로 약속합니다.

이러한 사고의 흐름을 따르면, 각 객체가 자신의 책임을 다하는 아래와 같은 코드가 자연스럽게 만들어집니다.

**[To-Be: 책임 중심 설계로 개선된 코드]**

```tsx
// 행동에 필요한 외부 정보를 담는 컨텍스트
interface ValidationContext {
  /* ... */
}

// 모든 쿠폰이 따라야 할 약속(인터페이스)
interface BaseCoupon {
  isValid(context: ValidationContext): boolean;
}

class FixedDiscountCoupon implements BaseCoupon {
  isValid(context: ValidationContext): boolean {
    // 자신의 유효성을 스스로 검증
  }
}

// 외부에서는 그저 약속된 행동을 '요청'할 뿐
function validateCoupons(coupon: BaseCoupon, context: ValidationContext): boolean {
  return coupon.isValid(context);
}
```

이제 `validateCoupons` 함수는 쿠폰의 종류를 묻지 않고 책임을 위임합니다. 그 결과 **개발자의 인지 부하가 낮아지고,** 변경에 유연한 코드가 되었습니다.

### 3.2. 컴포넌트의 '책임'은 명확한가요?

**'책임'** 이란 객체가 협력 속에서 수행해야 할 행동을 의미합니다. 이는 프론트엔드에서 익히 들어온 **단일 책임 원칙(Single Responsibility Principle, SRP)** 과 맞닿아 있습니다. 하나의 컴포넌트는 하나의 책임만 가져야 한다는 원칙입니다.

책임이 명확하지 않은 컴포넌트는 금세 비대해집니다. API 호출, 상태 관리, UI 렌더링, 이벤트 처리 등 온갖 로직이 뒤섞인 **거대 컴포넌트(Massive Component)** 는 변경의 파급 효과를 예측하기 어렵고, 재사용은 거의 불가능에 가깝습니다.

#### 책임 분리 예시: 데이터 '생성'과 '사용'의 분리

가장 흔하면서도 효과적인 책임 분리는 데이터 **'생성'(Fetching & State Management)의 책임**과 **'사용'(UI Rendering)의 책임**을 나누는 것입니다.

**[As-Is: 책임이 혼재된 거대 컴포넌트]**

```tsx
// CouponListComponent.tsx
function CouponListComponent() {
  const [coupons, setCoupons] = useState([]);
  const [isLoading, setIsLoading] = useState(true);

  // 1. 데이터 '생성' 책임 (API 호출)
  useEffect(() => {
    const fetchCoupons = async () => {
      try {
        const response = await fetch('/api/coupons');
        const data = await response.json();
        setCoupons(data);
      } catch (error) {
        console.error('쿠폰을 불러오는데 실패했습니다.', error);
      } finally {
        setIsLoading(false);
      }
    };
    fetchCoupons();
  }, []);

  // 2. 데이터 '사용' 책임 (UI 렌더링)
  if (isLoading) return <div>쿠폰 목록을 불러오는 중...</div>;
  return (
    <ul>
      {coupons.map((coupon) => (
        <li key={coupon.id}>{coupon.name}</li>
      ))}
    </ul>
  );
}
```

이 컴포넌트는 '사용 가능한 쿠폰 목록을 가져오는 방법'과 '쿠폰 목록을 화면에 그리는 방법'이라는 두 가지 책임을 모두 가지고 있습니다. 만약 API 주소를 변경하거나 로딩 UI를 바꾸고 싶을 때, 우리는 항상 같은 파일을 수정해야 합니다.

**[To-Be: 커스텀 훅으로 책임을 분리한 구조]**

커스텀 훅은 프론트엔드에서 책임을 분리하는 가장 강력하고 현대적인 방법 중 하나입니다.

```tsx
// useCoupons.ts (데이터 '생성' 책임)
function useCoupons() {
  const [coupons, setCoupons] = useState([]);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    const fetchCoupons = async () => {
      try {
        const response = await fetch('/api/coupons');
        const data = await response.json();
        setCoupons(data);
      } catch (error) {
        console.error('쿠폰을 불러오는데 실패했습니다.', error);
      } finally {
        setIsLoading(false);
      }
    };
    fetchCoupons();
  }, []);

  return { coupons, isLoading };
}

// CouponListComponent.tsx (데이터 '사용' 책임)
function CouponListComponent() {
  // 이제 이 컴포넌트는 데이터가 '어떻게' 오는지 모른다.
  // 오직 '무엇'({ coupons, isLoading })을 사용하는지만 안다.
  const { coupons, isLoading } = useCoupons();

  if (isLoading) return <div>쿠폰 목록을 불러오는 중...</div>;
  return (
    <ul>
      {coupons.map((coupon) => (
        <li key={coupon.id}>{coupon.name}</li>
      ))}
    </ul>
  );
}
```

`useCoupons` 훅은 데이터 생성이라는 단일 책임을, `CouponListComponent` 컴포넌트는 UI 렌더링이라는 단일 책임을 갖게 되었습니다. 이제 책임 분리가 가져다주는 유연성의 대표적인 예시를 다음 장에서 확인해 보겠습니다.

### 3.3. 컴포넌트에게 '역할'을 부여하면 생기는 일

**'역할'** 은 여러 컴포넌트가 공유할 수 있는 책임의 집합을 **추상화**한 것입니다. 컴포넌트가 구체적인 다른 '컴포넌트'에 의존하는 것이 아니라, 추상적인 '역할'에 의존하게 되면 시스템의 유연성과 재사용성이 극적으로 향상됩니다.

이는 **개방-폐쇄 원칙(Open-Closed Principle, OCP)** 과 깊은 관련이 있습니다. OCP는 "확장에는 열려 있고, 수정에는 닫혀 있어야 한다"는 원칙으로, 새로운 기능이 추가되더라도 기존의 안정적인 코드는 수정되지 않아야 함을 의미합니다.

#### 역할 부여 예시: CouponCard의 역할 분리

다양한 종류의 쿠폰을 보여주는 `CouponCard` 컴포넌트를 생각해 봅시다.

**[As-Is: 역할이 분리되지 않아 OCP를 위반하는 구조]**

```tsx
// CouponCard.tsx
// 이 컴포넌트는 '쿠폰 프레임' 역할과 '어떤 쿠폰을 보여줄지 결정'하는
// 두 가지 역할을 모두 수행하고 있다.
function CouponCard({ coupon }: { coupon: Coupon }) {
  const renderSpecificCouponDetails = () => {
    // 새로운 쿠폰 타입이 추가되면 이 코드를 '수정'해야 한다. (OCP 위반)
    switch (coupon.type) {
      case 'FIXED':
        return <FixedCouponDetails coupon={coupon} />;
      case 'PERCENTAGE':
        return <PercentageCouponDetails coupon={coupon} />;
      default:
        return null;
    }
  };

  return (
    <div className="card-frame">
      <CommonCouponInfo coupon={coupon} />
      {renderSpecificCouponDetails()}
    </div>
  );
}
```

`CouponCard`는 모든 종류의 쿠폰 상세 컴포넌트의 존재를 알아야 합니다. 새로운 쿠폰 타입이 추가될 때마다 이 컴포넌트의 `switch` 문은 변경되어야만 합니다. 안정적이어야 할 저수준 컴포넌트가 계속해서 수정에 노출되는 구조입니다.

**[To-Be: 역할 분리를 통해 OCP를 준수하는 구조]**

`CouponCard`에게는 **'쿠폰의 공통 프레임을 렌더링하는 역할'** 만 부여하고, **'어떤 종류의 쿠폰을 보여줄지 결정하는 역할'** 은 상위 컴포넌트로 위임합니다.

```tsx
// CouponCard.tsx (수정에 닫혀있게 됨)
// 이 컴포넌트의 역할은 오직 '주입받은 내용물(children)을 프레임 안에 보여주는 것'이다.
// 어떤 내용물이 들어올지는 전혀 신경 쓰지 않는다.
function CouponCard({ coupon, children }: CouponCardProps) {
  return (
    <div className="card-frame">
      <CommonCouponInfo coupon={coupon} />
      {children}
    </div>
  );
}

// CouponListPage.tsx (확장에 열려있게 됨)
// 상위 컴포넌트가 '결정'의 역할을 맡는다.
const COUPON_DETAILS_MAP = {
  FIXED: FixedCouponDetails,
  PERCENTAGE: PercentageCouponDetails,
  // 새로운 쿠폰 타입이 생기면 이 MAP에 한 줄 '추가'만 하면 된다.
};

function CouponListPage({ coupons }) {
  return coupons.map((coupon) => {
    const SpecificDetails = COUPON_DETAILS_MAP[coupon.type];
    return (
      <CouponCard
        key={coupon.id}
        coupon={coupon}
      >
        <SpecificDetails coupon={coupon} />
      </CouponCard>
    );
  });
}
```

이제 `CouponCard`는 `children`이라는 추상적인 역할에만 의존합니다. 새로운 쿠폰 타입이 생겨도 `CouponCard`는 절대 수정되지 않습니다. 대신, 변경이 발생할 가능성이 높은 `CouponListPage`의 설정(`COUPON_DETAILS_MAP`)에 한 줄을 추가하기만 하면 됩니다. 이것이 바로 역할을 부여하여 유연성을 확보하고 OCP를 준수하는 설계입니다.

## 4\. 프론트엔드 코드 예시로 알아보기: 커스텀 훅과 SOLID

앞서 살펴본 커스텀 훅 분리 예시는 객체지향 설계 원칙, 특히 SOLID 원칙이 어떻게 프론트엔드 코드의 품질을 높이는지 명확하게 보여줍니다.

**커스텀 훅 분리는 SOLID 중 어떤 원칙에 해당할까요?**

1.  **단일 책임 원칙 (SRP):**

    - **훅:** 데이터 페칭, 상태 관리라는 **'하나의 책임'** 을 진다.
    - **컴포넌트:** 데이터를 받아 UI를 렌더링하는 **'하나의 책임'** 을 진다.

2.  **의존성 역전 원칙 (DIP - Dependency Inversion Principle):**

    - "상위 모듈은 하위 모듈에 의존해서는 안 된다. 둘 모두 추상화에 의존해야 한다."
    - 이 개념을 더 쉽게 이해하기 위해 **'노트북, 콘센트, 발전소'** 비유를 들어보겠습니다.

> **[나쁜 설계]** 만약 우리가 사용하는 **노트북**에 전원 어댑터(플러그)가 없고, **수력 발전소까지 직접 전선이 연결**되어 있다고 상상해봅시다. 노트북은 잘 작동하겠지만, 만약 전력 회사가 발전소를 원자력 발전소로 교체한다면 큰일입니다. 우리는 노트북과 연결된 전선을 **모두 뜯어내고 원자력 발전소까지 새로 연결**해야 합니다.
>
> **[좋은 설계]** 이 문제를 해결하는 것이 바로 **'콘센트', 즉 인터페이스**입니다. 우리의 노트북은 표준 규격의 어댑터를 이용해 **콘센트**에만 연결됩니다. 덕분에 전력 회사가 내부적으로 발전 방식을 수력에서 원자력으로, 또 태양광으로 바꾸더라도, 우리는 하던 대로 콘센트를 그대로 쓰면 됩니다. **노트북**은 바꿀 필요가 전혀 없죠.

이 비유를 우리 코드에 적용해 봅시다.

- **노트북 (상위 모듈):** `CouponListComponent`. 이 컴포넌트의 목표는 '쿠폰 목록을 화면에 표시하는 것'입니다.
- **발전소 (하위 모듈):** `fetch` 또는 `tanstack-query` 같은 실제 데이터 fetching 구현 기술입니다.
- **콘센트 (인터페이스/추상화):** `useCoupons` 훅이 반환하는 `{ coupons, isLoading }` 라는 표준 약속입니다.

**[DIP 위반: '노트북'이 '발전소'에 직접 연결된 경우]**

`useCoupons` 훅 없이 컴포넌트가 `fetch` 로직을 직접 포함하고 있다면, 이는 '노트북이 발전소에 직접 연결된 것'과 같습니다. `fetch`를 `tanstack-query`로 바꾸려면 노트북(컴포넌트) 자체를 뜯어고쳐야 합니다.

**[DIP 준수: '노트북'이 '콘센트'에 연결된 경우]**

`useCoupons`라는 '콘센트'를 중간에 둠으로써, `CouponListComponent`('노트북')는 오직 `{ coupons, isLoading }`이라는 표준 규격에만 의존하게 됩니다.

이러한 DIP 구조 덕분에 우리는 놀라운 유연성을 확보합니다. 프로젝트의 '발전소'를 `tanstack-query`로 변경하기로 결정했다고 가정해 봅시다.

**[As-Is: '수력 발전소'(`fetch`)가 연결된 '콘센트']**

```tsx
// useCoupons.ts
function useCoupons() {
  const [coupons, setCoupons] = useState([]);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    // ... fetch 로직 (수력 발전소) ...
  }, []);

  // '콘센트 규격'을 제공
  return { coupons, isLoading };
}
```

**[To-Be: '원자력 발전소'(`tanstack-query`)로 교체된 '콘센트']**

```tsx
// useCoupons.ts (tanstack-query로 리팩토링)
import { useQuery } from '@tanstack/react-query';

const fetchCouponsAPI = async () => {
  /* ... */
};

function useCoupons() {
  // ... tanstack-query 로직 (원자력 발전소) ...
  const { data, isLoading } = useQuery({
    /* ... */
  });

  // 컴포넌트가 의존하는 '콘센트 규격'을 그대로 유지
  return { coupons: data || [], isLoading };
}
```

`useCoupons` 훅의 내부 '발전소'는 완전히 바뀌었지만, 훅이 반환하는 **'콘센트 규격'** 은 그대로 유지됩니다. 덕분에 우리는 이 훅을 사용하는 **수십 개의 '노트북'(컴포넌트) 코드를 단 한 줄도 변경할 필요가 없습니다.** 이것이 바로 협력, 책임, 역할의 원칙을 기반으로 한 추상화가 우리에게 주는 실질적인 이점입니다.

## 5\. 주의할 점

> **⚠️ 이 섹션은 특히 중요합니다!**
>
> 지금까지 소개한 원칙들은 강력한 도구이지만, 잘못 사용하면 오히려 코드를 복잡하게 만들 수 있습니다. 아래 두 가지 원칙을 꼭 기억하세요.

### 💡 핵심 1: 원칙은 법칙이 아니다 - 섣부른 추상화의 위험

유연한 설계는 공짜로 얻어지지 않습니다. 추상화 계층이 늘어날수록 코드의 복잡도는 증가하고, 전체 구조를 파악하기 어려워질 수 있습니다. 아직 일어나지도 않은 불확실한 변경을 예측하여 과도한 설계를 적용하는 것은 '오버 엔지니어링'일 뿐입니다.

> "세 번의 법칙" 에 따라, 특정 로직이나 패턴이 세 번 이상 반복되기 전까지는 섣불리 추상화하지 않는 것이 현명합니다. 처음에는 가장 단순하고 직접적인 방법으로 구현하고, 구체적인 요구사항과 패턴이 명확해졌을 때 리팩토링을 통해 추상화하는 것이 현명합니다.

### 🎯 핵심 2: 어디에 집중해야 하는가? - 경계면(Seams)에 집중하기

모든 코드를 유연하게 만들 필요는 없습니다. 대신, 변경이 자주 발생할 것으로 예상되는 시스템의 중요한 **'경계면'** 에 설계 원칙을 집중적으로 적용하는 것이 효과적입니다.

**1) UI와 비즈니스 로직의 경계**

이는 '보여주는 것'과 '계산하거나 판단하는 것'을 나누는 것입니다. 예를 들어:

```tsx
// ❌ 나쁜 예: UI 컴포넌트가 비즈니스 로직을 직접 포함
function ProductCard({ product }) {
  // 할인 계산 로직이 컴포넌트 안에 있음
  const discountedPrice = product.price * 0.8;
  const isCheap = discountedPrice < 10000;

  return (
    <div>
      <h3>{product.name}</h3>
      <p>{discountedPrice}원</p>
      {isCheap && <span>저렴해요!</span>}
    </div>
  );
}

// ✅ 좋은 예: 비즈니스 로직을 분리
// useProductPrice.ts - 비즈니스 로직의 책임
function useProductPrice(product) {
  const discountedPrice = product.price * 0.8;
  const isCheap = discountedPrice < 10000;

  return { discountedPrice, isCheap };
}

// ProductCard.tsx - UI 렌더링의 책임
function ProductCard({ product }) {
  const { discountedPrice, isCheap } = useProductPrice(product);

  return (
    <div>
      <h3>{product.name}</h3>
      <p>{discountedPrice}원</p>
      {isCheap && <span>저렴해요!</span>}
    </div>
  );
}
```

이렇게 분리하면 할인율 정책이 변경되거나 "저렴함"의 기준이 바뀌어도, UI 컴포넌트는 전혀 수정할 필요가 없습니다.

**2) 애플리케이션과 외부 서비스의 경계**

UI 컴포넌트가 API 호출 책임까지 떠안지 않도록, **커스텀 훅으로 데이터 페칭을 감싸** 외부 서비스와의 경계를 만든다.

```tsx
// ❌ 나쁜 예: 컴포넌트 안에서 바로 fetch
function UserProfile() {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetch('https://api.example.com/v1/users/me')
      .then((res) => res.json())
      .then((data) => setUser(data));
  }, []);

  return <div>{user?.name}</div>;
}

// ✅ 좋은 예: 커스텀 훅이 외부 서비스와의 경계를 담당
function useCurrentUser() {
  const [user, setUser] = useState(null);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    const fetchUser = async () => {
      const response = await fetch('https://api.example.com/v1/users/me');
      const data = await response.json();
      setUser(data);
      setIsLoading(false);
    };

    fetchUser();
  }, []);

  return { user, isLoading };
}

function UserProfile() {
  const { user, isLoading } = useCurrentUser();

  if (isLoading) return <p>불러오는 중...</p>;
  return <div>{user?.name}</div>;
}
```

커스텀 훅이 API URL·에러 처리·로딩 상태를 책임지기 때문에, 컴포넌트는 "무엇을 보여줄지"에만 집중할 수 있습니다. 추후 API 버전이 바뀌거나 인증 토큰 처리가 필요해져도 `useCurrentUser`만 수정하면 됩니다.

**3) 애플리케이션과 브라우저 API의 경계**

`localStorage`, `fetch` 등을 직접 사용하기보다, 이를 감싸는 래퍼(Wrapper) 함수를 만들어 의존성을 관리합니다.

```tsx
// ❌ 나쁜 예: 컴포넌트가 localStorage를 직접 사용
function ThemeToggle() {
  const [theme, setTheme] = useState(() => {
    return localStorage.getItem('theme') || 'light';
  });

  const toggleTheme = () => {
    const newTheme = theme === 'light' ? 'dark' : 'light';
    localStorage.setItem('theme', newTheme);
    setTheme(newTheme);
  };

  return <button onClick={toggleTheme}>테마 변경</button>;
}

// ✅ 좋은 예: 브라우저 API를 래퍼로 감싸기
// storage.ts - 브라우저 API와의 경계
const storage = {
  get: (key: string) => {
    try {
      return localStorage.getItem(key);
    } catch {
      // localStorage 접근 불가 시 대체 로직
      return null;
    }
  },
  set: (key: string, value: string) => {
    try {
      localStorage.setItem(key, value);
    } catch {
      // 에러 처리
    }
  },
};

// useTheme.ts
function useTheme() {
  const [theme, setTheme] = useState(() => {
    return storage.get('theme') || 'light';
  });

  const toggleTheme = () => {
    const newTheme = theme === 'light' ? 'dark' : 'light';
    storage.set('theme', newTheme);
    setTheme(newTheme);
  };

  return { theme, toggleTheme };
}

// ThemeToggle.tsx
function ThemeToggle() {
  const { theme, toggleTheme } = useTheme();
  return <button onClick={toggleTheme}>테마 변경</button>;
}
```

이렇게 분리하면 `localStorage`를 `sessionStorage`나 서버 저장소로 바꾸거나, 암호화 로직을 추가할 때도 `storage` 모듈만 수정하면 됩니다.

근본적인 설계 원칙에 대한 이해 없이 단순히 최신 기술을 사용하는 것은 모래 위에 성을 쌓는 것과 같습니다. 기술의 변화 속에서도 흔들리지 않는 단단한 설계의 기반을 다지는 것이 무엇보다 중요한 것 같습니다.

## 6\. 마치며

이 글을 통해 우리는 객체지향의 핵심 원칙인 **협력, 책임, 역할**이 프론트엔드 개발, 특히 컴포넌트 설계에 어떻게 강력한 통찰을 제공하는지 살펴보았습니다.

설계의 출발점을 '어떤 데이터가 필요한가'에서 **'컴포넌트들이 어떻게 협력해야 하는가'** 로 바꾸는 작은 관점의 전환이 코드의 구조를 완전히 바꿀 수 있습니다.

- **협력** 관계를 정의하면,
- 각 컴포넌트의 **책임**이 명확해지고,
- 명확해진 책임들은 추상화된 **역할**이 되어
- 시스템 전체의 유연성과 재사용성을 높입니다.

이 원칙들을 당장 모든 코드에 적용하기는 어려울 수 있습니다. 하지만 다음 컴포넌트를 만들기 전, "이 컴포넌트의 가장 중요한 책임은 무엇일까?", "누구와 어떻게 협력해야 할까?"라는 질문을 던져보는 것만으로도 충분합니다.
