# 서버 롤백 시 데이터베이스 스키마 불일치 문제 해결

## 1. 문제 발견

### 1.1 배포 전략과 롤백 방식

운영 서버를 배포하는 과정에서 잘못된 배포에 대비한 롤백 방법을 도입했습니다. Docker 이미지를 활용해 서버가 중단되면 GitHub Actions를 통해 이전 릴리즈 버전으로 되돌리는 전략이었습니다.

배포 시점에 서버와 데이터베이스를 동일한 버전으로 맞춰 배포했습니다. 새로운 기능이 추가되면 서버 코드와 데이터베이스 스키마를 함께 변경하고, 이를 하나의 릴리즈로 묶어 배포했습니다.

### 1.2 문제 발생 시나리오

새로운 릴리즈에서 nullable 속성을 가진 컬럼을 추가했습니다. 배포는 정상적으로 완료되었지만, 서버에 예상치 못한 오류가 발생해 서비스가 중단되었습니다. 이때 GitHub Actions를 통해 서버를 이전 버전으로 되돌렸습니다.

서버는 이전 버전으로 돌아갔지만, 데이터베이스는 최신 스키마를 유지했습니다. 롤백된 서버 코드는 새로 추가된 컬럼의 존재를 알지 못했고, 데이터 삽입 시 다음과 같은 오류가 발생했습니다.

```
Field 'column_name' doesn't have a default value
```

이전 버전의 서버 코드는 새 컬럼에 값을 지정하지 않았고, 데이터베이스는 해당 컬럼에 기본값이 없어 오류를 반환했습니다.

### 1.3 문제의 근본 원인

문제의 핵심은 서버와 데이터베이스의 버전을 분리하지 않았다는 점입니다.

배포 시점에는 서버와 데이터베이스를 같은 버전으로 맞춰 배포했습니다. 하지만 롤백 시점에는 서버만 이전 버전으로 되돌아갔고, 데이터베이스는 최신 상태를 유지했습니다. 이러한 버전 불일치가 스키마와 서버 코드 간 충돌을 일으켰습니다.

추가로 분석한 결과, 다음과 같은 상황에서도 유사한 문제가 발생할 수 있음을 확인했습니다.

- 컬럼명을 변경했을 때, 롤백된 서버가 이전 컬럼명으로 접근하려 하면 오류가 발생합니다.
- 컬럼을 삭제했을 때, 롤백된 서버가 삭제된 컬럼에 접근하려 하면 오류가 발생합니다.
- 제약 조건(NOT NULL, UNIQUE 등)을 추가했을 때, 롤백된 서버가 제약 조건을 위반하는 데이터를 삽입하려 하면 오류가 발생합니다.

## 2. 해결 방안 탐색

### 2.1 요구사항 정의

문제를 해결하기 위해 다음 세 가지 요구사항을 정의했습니다.

첫째, 서버를 롤백해도 데이터베이스와 호환되어야 합니다. 롤백 전후 버전의 서버가 모두 정상적으로 동작해야 합니다.

둘째, 롤백 전에 저장된 데이터가 유지되어야 합니다. 데이터 손실 없이 서비스를 복구할 수 있어야 합니다.

셋째, 비용만으로 문제를 해결하지 않아야 합니다. 인프라를 추가하는 방식이 아닌, 설계와 전략으로 해결해야 합니다.

### 2.2 첫 번째 방안: Binlog 기반 시점 복구

데이터베이스 롤백 방법을 조사하던 중 Binlog(Binary Log) 기반 PITR(Point-In-Time Recovery) 방식을 발견했습니다.

Binlog는 데이터베이스의 모든 변경 사항을 시간 순서대로 기록하는 로그 파일입니다. PITR은 이 로그를 활용해 특정 시점의 데이터베이스 상태로 복구하는 방식입니다.

백업 시점의 데이터베이스를 복원하고, Binlog를 재생해 원하는 시점까지 데이터를 복구합니다. 예를 들어, 오전 9시에 백업했고 오후 3시에 문제가 발생했다면, 오전 9시 백업본을 복원한 후 오후 2시 59분까지의 Binlog를 재생해 문제 발생 직전 상태로 되돌릴 수 있습니다.

하지만 이 방식은 다음과 같은 한계가 있었습니다.

첫째, 심각한 장애 상황에서만 사용하는 최후의 수단입니다. 데이터베이스를 완전히 재구축해야 하는 상황, 즉 데이터가 손상되거나 삭제된 경우에 적합합니다.

둘째, 복구 과정에서 데이터 손실 위험이 있습니다. 마지막 백업 시점 이후의 Binlog가 손상되거나 유실되면, 해당 시간대의 데이터를 복구할 수 없습니다.

셋째, 복구에 상당한 시간이 필요합니다. 백업본 복원과 Binlog 재생 과정에서 서비스 중단 시간이 길어질 수 있습니다.

이러한 이유로 Binlog 기반 방식은 요구사항에 적합하지 않다고 판단했습니다.

### 2.3 두 번째 방안: 데이터베이스 이중화

MySQL 인스턴스를 두 개 운영하는 방안을 검토했습니다. 서버 배포 전략 중 Blue-Green 배포나 Canary 배포처럼, 데이터베이스도 두 개를 운영하면 서버 롤백 시 이전 데이터베이스를 사용할 수 있습니다.

Blue-Green 배포는 두 개의 동일한 환경을 준비하고, 트래픽을 전환하는 방식입니다. 새 버전(Green)을 배포하고 테스트한 후, 문제가 없으면 트래픽을 이전 버전(Blue)에서 새 버전으로 전환합니다. 문제가 발생하면 즉시 Blue 환경으로 되돌릴 수 있습니다.

이 방식을 데이터베이스에 적용하면 다음과 같이 동작합니다.

새 릴리즈 배포 시, 새로운 데이터베이스 인스턴스를 시작하고 스키마를 변경합니다. 새 서버는 새 데이터베이스를 사용하고, 이전 서버는 이전 데이터베이스를 계속 사용합니다. 배포가 성공하면 트래픽을 새 서버로 전환하고, 롤백이 필요하면 이전 서버와 데이터베이스로 되돌립니다.

하지만 이 방식은 다음과 같은 문제가 있었습니다.

첫째, 비용이 2배로 증가합니다. 데이터베이스 인스턴스를 항상 두 개 운영해야 하므로, 서버 비용과 스토리지 비용이 2배가 됩니다.

둘째, 데이터 동기화 문제가 발생합니다. 새 데이터베이스에 저장된 데이터를 이전 데이터베이스로 되돌릴 때, 데이터 정합성을 보장하기 어렵습니다.

셋째, 복잡도가 크게 증가합니다. 두 데이터베이스 간 데이터 동기화, 트래픽 전환, 롤백 시나리오 등을 모두 관리해야 합니다.

넷째, 세 번째 요구사항을 위배합니다. 비용만으로 문제를 해결하는 방식이므로, 근본적인 해결책이 아닙니다.

이러한 이유로 데이터베이스 이중화 방안도 적절하지 않다고 판단했습니다.

### 2.4 세 번째 방안: 하위 호환성 확보

두 가지 방안을 검토하면서 근본적인 질문을 던지게 되었습니다.

"데이터베이스와 서버가 반드시 같은 버전을 사용해야 할까?"

데이터베이스의 유연성은 상당히 높습니다. 컬럼에 nullable 속성을 적절히 부여하면, 하나의 테이블로 여러 유형의 데이터를 저장할 수 있습니다. 예를 들어, 학교 구성원 테이블에 선생님, 학생, 주무관의 정보를 모두 저장할 수 있습니다. 선생님에게만 필요한 컬럼은 학생 데이터에서 null로 남겨두고, 학생에게만 필요한 컬럼은 선생님 데이터에서 null로 남겨둡니다.

이러한 특성을 활용하면, 데이터베이스 스키마를 롤백 전후 버전 모두와 호환되도록 만들 수 있습니다. 즉, 이전 버전의 서버와 최신 버전의 서버가 모두 같은 데이터베이스 스키마를 사용할 수 있습니다.

이 접근 방식의 핵심은 "스키마를 점진적으로 변경한다"는 것입니다. 한 번에 큰 변경을 적용하지 않고, 여러 단계로 나누어 변경합니다. 각 단계에서 이전 버전과 새 버전 모두 정상적으로 동작하도록 보장합니다.

관련 자료를 조사하던 중, Prisma의 문서에서 "Expand and Contract Pattern"이라는 전략을 발견했습니다. 이 전략이 요구사항을 충족하는 해결책임을 확인했습니다.

## 3. Expand and Contract Pattern

### 3.1 전략 개요

Expand and Contract Pattern은 데이터베이스 스키마 변경을 세 단계로 나누어 진행하는 전략입니다.

첫 번째 단계는 확장(Expand)입니다. 기존 스키마에 새로운 요소를 추가하되, 기존 기능에 영향을 주지 않도록 합니다. 예를 들어, 새 컬럼을 추가할 때 nullable로 만들어 기존 코드가 계속 동작하도록 합니다.

두 번째 단계는 마이그레이션(Migration)입니다. 애플리케이션 코드를 새 스키마에 맞춰 변경합니다. 이전 버전의 서버와 새 버전의 서버가 공존할 수 있도록, 두 방식을 모두 지원하는 코드를 작성합니다.

세 번째 단계는 축소(Contract)입니다. 충분한 시간이 지나 모든 서버가 새 버전으로 배포되었다면, 이전 스키마 요소를 제거합니다. 예를 들어, 컬럼명을 변경했다면 이전 컬럼을 삭제합니다.

이 전략의 핵심은 "스키마를 항상 양쪽 버전에서 읽고 쓸 수 있는 상태로 유지한다"는 것입니다. 확장과 마이그레이션 단계에서는 이전 버전과 새 버전 모두 정상적으로 동작합니다. 롤백이 발생하더라도 데이터베이스 스키마는 두 버전을 모두 지원하므로, 오류가 발생하지 않습니다.

### 3.2 전략의 장점

첫째, 서버 롤백 시 안정성이 보장됩니다. 데이터베이스 스키마가 이전 버전과 새 버전을 모두 지원하므로, 어느 버전으로 롤백하더라도 정상적으로 동작합니다.

둘째, 무중단 배포가 가능합니다. 스키마 변경과 서버 배포를 분리할 수 있어, 서비스를 중단하지 않고 배포할 수 있습니다.

셋째, 배포 위험이 감소합니다. 스키마 변경을 단계별로 나누어 진행하므로, 각 단계의 위험을 최소화할 수 있습니다.

넷째, 추가 인프라 비용이 발생하지 않습니다. 기존 데이터베이스를 그대로 사용하며, 스키마 설계 방식만 변경하면 됩니다.

다섯째, 데이터 손실 위험이 없습니다. 기존 데이터를 그대로 유지하면서 스키마를 변경하므로, 데이터가 손실되거나 손상될 위험이 없습니다.

### 3.3 전략 적용 원칙

Expand and Contract Pattern을 적용할 때 다음 원칙을 준수해야 합니다.

첫째, 새로 추가하는 컬럼은 반드시 nullable로 만듭니다. 이전 버전의 서버는 새 컬럼을 알지 못하므로, 값을 지정하지 않습니다. nullable이 아니면 기본값이 없어 오류가 발생합니다.

둘째, 제약 조건은 최종 단계에서 추가합니다. NOT NULL, UNIQUE, FOREIGN KEY 같은 제약 조건은 이전 버전의 서버와 충돌할 수 있으므로, 모든 서버가 새 버전으로 배포된 후에 추가합니다.

셋째, 컬럼을 즉시 삭제하지 않습니다. 컬럼을 삭제하면 이전 버전의 서버에서 오류가 발생합니다. 충분한 시간이 지나 롤백 가능성이 없어진 후에 삭제합니다.

넷째, 데이터 마이그레이션은 점진적으로 수행합니다. 대량의 데이터를 한 번에 변경하면 데이터베이스 잠금이 발생할 수 있습니다. 배치 작업으로 나누어 천천히 수행합니다.

다섯째, 충분한 안정화 기간을 확보합니다. 새 버전 배포 후 최소 2~3회의 릴리즈 주기가 지나야 이전 요소를 제거합니다. 이 기간 동안 롤백이 발생하지 않으면, 안전하게 이전 요소를 제거할 수 있습니다.

## 4. 스키마 변경 시나리오별 적용 방법

### 4.1 컬럼 추가

새로운 컬럼을 추가하는 경우는 비교적 간단합니다.

**단계 1: 컬럼 추가 및 코드 변경 (릴리즈 N)**

새 컬럼을 nullable로 추가합니다.

```sql
ALTER TABLE device
ADD COLUMN device_type VARCHAR(50);
```

이 시점에서 이전 버전의 서버는 `device_type` 컬럼을 사용하지 않습니다. 새로 생성되는 행에는 `device_type`이 null로 저장됩니다.

서버 코드를 변경해 `device_type` 컬럼을 사용하도록 합니다.

만약 이 시점에서 롤백이 발생한다고 하더라도, 이전 서버는 `device_type` 을 모르기 때문에, 서비스가 올바르게 동작합니다.

**단계 2: 제약 조건 추가 (릴리즈 N+3)**

필요하다면 제약 조건을 추가합니다.

```sql
ALTER TABLE device
MODIFY COLUMN device_type VARCHAR(50) NOT NULL DEFAULT 'UNKNOWN';
```

### 4.2 컬럼명 변경

컬럼명을 변경하는 경우는 복잡합니다. 기존 컬럼과 새 컬럼을 동시에 유지해야 합니다.

**예시: `device_id` → `device_uuid` 변경**

**단계 1: 기존 컬럼 nullable 처리 (릴리즈 N)**

기존 컬럼을 nullable로 변경합니다.

```sql
ALTER TABLE device
MODIFY COLUMN device_id VARCHAR(50);
```

이 시점에서 이전 버전의 서버는 여전히 `device_id`를 사용합니다.

**단계 2: 새 컬럼 추가 (릴리즈 N)**

새 컬럼을 nullable로 추가합니다.

```sql
ALTER TABLE device
ADD COLUMN device_uuid VARCHAR(255);
```

**단계 3: 코드 변경 및 이중 쓰기 (릴리즈 N)**

서버 코드를 변경해 `device_uuid`를 사용하도록 합니다.

```java
// 새로운 데이터 저장 시
String uuid = UUID.randomUUID().toString();
device.setDeviceId(uuid); // 제거
device.setDeviceUuid(uuid);
```

**단계 4: 기존 데이터 마이그레이션 (릴리즈 N+2)**

기존 데이터의 `device_uuid`를 채웁니다. 대량 데이터라면 배치 작업으로 나누어 수행합니다.

```sql
UPDATE device
SET device_uuid = device_id
WHERE device_uuid IS NULL
LIMIT 1000;
```

이 작업을 `device_uuid`가 null인 행이 없을 때까지 반복합니다.

**단계 5: 제약 조건 추가 (릴리즈 N+3)**

모든 데이터 마이그레이션이 완료되면, `device_uuid`에 제약 조건을 추가합니다.

```sql
ALTER TABLE device
MODIFY COLUMN device_uuid VARCHAR(255) NOT NULL;
```

**단계 6: 기존 컬럼 삭제 (릴리즈 N+5)**

충분한 시간이 지나 롤백 가능성이 없어지면, `device_id` 컬럼을 삭제합니다.

```sql
ALTER TABLE device
DROP COLUMN device_id;
```

### 4.3 컬럼 삭제

컬럼을 삭제하는 경우도 신중하게 진행해야 합니다.

**예시: `legacy_field` 컬럼 삭제**

**단계 1: 코드에서 사용 중단 (릴리즈 N)**

서버 코드에서 해당 컬럼을 사용하지 않도록 변경합니다. 읽기와 쓰기를 모두 제거합니다.

```java
// device.setLegacyField() 제거
// device.getLegacyField() 제거
```

**단계 2: 안정화 기간 (릴리즈 N+1 ~ N+2)**

최소 2~3회의 릴리즈 주기 동안 대기합니다. 이 기간 동안 롤백이 발생하지 않으면, 해당 컬럼이 더 이상 필요하지 않다고 판단할 수 있습니다.

**단계 3: 컬럼 삭제 (릴리즈 N+3)**

컬럼을 삭제합니다.

```sql
ALTER TABLE device
DROP COLUMN legacy_field;
```

참고로, 즉시 삭제하지 않고 nullable로 변경하는 방법도 있습니다.

**대안 단계 2: nullable 처리 (릴리즈 N)**

```sql
ALTER TABLE device
MODIFY COLUMN legacy_field VARCHAR(255);
```

이 방식은 삭제보다 안전하지만, 불필요한 컬럼이 남아있어 스키마가 복잡해질 수 있습니다.

추가적으로, 다양한 데이터베이스 변경이 있지만 앞서 설명한 세 가지를 이용한다면 다양한 상황에 적용이 가능하여 생략하겠습니다.

## 5. 결론

### 5.1 핵심 교훈

서버 롤백 시 데이터베이스 스키마 불일치 문제를 해결하면서 다음과 같은 교훈을 얻었습니다.

첫째, 서버와 데이터베이스는 절대 같은 버전을 유지할 수 없습니다. 배포 과정에서 시간 차이가 발생하고, 롤백 시 버전이 달라집니다. 이를 인정하고 설계에 반영해야 합니다.

둘째, 하위 호환성이 안정성의 핵심입니다. 데이터베이스 스키마가 여러 버전의 서버를 지원하면, 배포와 롤백이 안전해집니다.

셋째, 점진적 변경이 위험을 줄입니다. 큰 변경을 작은 단계로 나누면, 각 단계의 위험을 최소화하고 문제 발생 시 빠르게 대응할 수 있습니다.

넷째, 문서화와 공유가 성공의 열쇠입니다. 팀 전체가 변경 계획을 이해하고, 각자의 역할을 명확히 알아야 합니다.

다섯째, 모니터링이 신뢰를 만듭니다. 각 단계를 모니터링하고 검증하면, 다음 단계로 안전하게 진행할 수 있습니다.

### 5.2 적용 효과

Expand and Contract Pattern을 도입한 후 다음과 같은 효과를 얻었습니다.

**배포 안정성 향상**

패턴 도입 전에는 스키마 변경이 포함된 배포에서 롤백이 불가능했습니다. 도입 후에는 언제든지 안전하게 롤백할 수 있게 되었습니다. 실제로 1.1.1 버전에서 발생한 긴급 롤백을 문제없이 처리했습니다.

**서비스 중단 시간 감소**

스키마 변경으로 인한 서비스 중단이 없어졌습니다. 데이터베이스와 서버를 독립적으로 배포할 수 있어, 무중단 배포가 가능해졌습니다.

**데이터 손실 방지**

롤백 시 데이터 손실 위험이 사라졌습니다. 데이터베이스 스키마는 항상 최신 상태를 유지하므로, 어느 버전의 서버를 사용하더라도 모든 데이터에 접근할 수 있습니다.

**팀 협업 개선**

명확한 가이드라인과 체크리스트를 통해 팀원 모두가 스키마 변경 과정을 이해하게 되었습니다. 코드 리뷰에서 스키마 관련 문제를 사전에 발견하는 빈도가 증가했습니다.

**개발 속도 향상**

스키마 변경에 대한 두려움이 줄어들면서, 필요한 변경을 주저 없이 진행할 수 있게 되었습니다. 이전에는 스키마 변경을 최대한 미루려 했지만, 이제는 적절한 시기에 변경을 적용합니다.

### 5.3 남은 과제

Expand and Contract Pattern을 적용하면서 여전히 해결해야 할 과제도 발견했습니다.

**복잡도 증가**

단순한 스키마 변경도 여러 단계를 거쳐야 하므로, 전체 과정이 복잡해졌습니다. 팀원들이 이 과정에 익숙해지는 데 시간이 필요했습니다.

**릴리즈 주기 고려**

충분한 안정화 기간을 확보하려면 최소 5~6회의 릴리즈가 필요합니다. 릴리즈 주기가 짧다면 문제없지만, 주기가 길다면 전체 변경 과정이 몇 개월씩 소요될 수 있습니다.

**데이터 마이그레이션 자동화**

대용량 데이터 마이그레이션을 매번 수동으로 구현하는 것은 비효율적입니다. 범용적으로 사용할 수 있는 마이그레이션 도구를 개발할 필요가 있습니다.

### 5.4 마치며

운영 환경에서 서버를 안전하게 배포하고 롤백하는 것은 모든 개발 팀의 숙제입니다. 이 과정에서 데이터베이스 스키마는 종종 간과되지만, 실제로는 가장 중요한 요소입니다.

Expand and Contract Pattern은 이 문제에 대한 명확한 해답을 제시합니다. 스키마를 점진적으로 변경하고, 항상 여러 버전과 호환 가능한 상태로 유지하면, 배포와 롤백이 안전해집니다.

이 전략을 도입하는 것은 단순히 기술적인 해결책을 적용하는 것을 넘어, 팀의 배포 문화를 개선하는 과정입니다. 실패를 두려워하지 않고 빠르게 시도하고 배울 수 있는 환경을 만드는 것이 궁극적인 목표입니다.

## 7. 요약

Expand and Contract Pattern은 서버 롤백 시 발생하는 데이터베이스 스키마 불일치 문제를 해결하는 효과적인 전략입니다.

**핵심 원칙**
- 스키마를 항상 여러 버전과 호환 가능하게 유지합니다
- 변경을 Expand, Migrate, Contract 세 단계로 나눕니다
- 충분한 안정화 기간을 확보합니다

**주요 이점**
- 안전한 롤백이 가능합니다
- 서비스 중단 없이 배포할 수 있습니다
- 데이터 손실 위험이 없습니다

**적용 방법**
- 새 컬럼은 nullable로 추가합니다
- 이중 쓰기로 하위 호환성을 유지합니다
- 단계적으로 제약 조건을 추가하고 이전 요소를 제거합니다

이 전략을 도입하면 배포 안정성이 크게 향상되고, 팀은 더 자신 있게 변화를 시도할 수 있습니다.
