이번 글에서는 커피빵 프로젝트에서 버튼 인터랙션을 도입한 과정을 설계적인 관점에서 설명 할 예정입니다.

왜 버튼 인터랙션을 도입하고 싶었는가
커피빵은 커피 내기를 위한 플랫폼으로, 주요 사용 환경은 모바일입니다. 따라서 모바일 뷰에서의 사용자 경험이 중요했습니다.

초기에는 버튼에 :active 속성을 적용해 터치 시 배경색이 변하는 단순한 피드백만 제공하고 있었습니다. 그러나 :active로 효과를 적용할 경우 버튼을 누르고 있는 짧은 순간에만 효과가 적용됩니다. 때문에 실제 모바일 사용환경에서는 버튼 클릭에 대한 피드백이 충분히 전달되지 않는다는 문제가 있었습니다. 


 

저는 사용자가 버튼을 눌렀을 때 확실한 피드백을 주고 싶었고, 그래서 더 생동감있고 자연스러운 인터랙션을 넣고 싶었습니다. 이런 효과는 단순 css 만으로도 구현할 수 있지만, 코드를 더 깔끔하게 관리하고 내가 만든 효과를 다른 팀원들이 쉽게 사용할 수 있도록 만들기 위해서는 여러 가지를 고려해야 했습니다.

 

이런 점들을 고민하면서 구현을 진행하다 보니 세 가지 주요 문제에 부딪혔습니다.

중복된 CSS 코드의 관리 문제
단순 불리언 상태 관리의 한계
반복되는 상태 로직의 복잡성
이제 이 세 가지 문제를 어떻게 해결하며 구조를 발전시켜 나갔는지 정리해 보겠습니다. 

초기 구현 문제 및 리팩터링 과정
모든 코드 예시는 emotion을 사용하였습니다.

문제1 : 중복된 css 코드의 관리 문제
기존 터치 효과 CSS가 적용된 컴포넌트는 총 3개였습니다.

(1) 공통 버튼 컴포넌트, (2) RoomActionButton, (3) TextButton

 

각 컴포넌트의 styled 파일에 아래와 같은 코드를 각각 작성했습니다.
```css
&:active {
  background-color: #f53e41  // 배경색보다 더 진한 color;
}
components/
├── Button/
│   ├── Button.tsx
│   └── Button.styled.ts <- active 효과 작성 
│
├── RoomActionButton/
│   ├── RoomActionButton.tsx
│   └── RoomActionButton.styled.ts <- active 효과 작성 
│
└── TextButton/
    ├── TextButton.tsx
    └── TextButton.styled.ts <- active 효과 작성
 
```

이 방식의 가장 큰 문제는 애니메이션 효과를 수정하거나 추가할 때마다, 모든 컴포넌트의 스타일 파일을 일일이 수정해야 한다는 점입니다. 결국 동일한 코드가 여러 곳에 중복되면서 관리가 어려워지고, 그에 따라 유지보수 비용도 점점 증가했습니다.

 

이러한 문제를 겪으며 버튼 효과를 한곳에서 관리하고 쉽게 수정할 수 있는 구조가 필요하다고 느꼈습니다. 그래서 디자인 시스템 관점에서, 여러 컴포넌트에 일관된 인터랙션을 적용할 수 있는 방법을 고민하게 되었습니다.

 

해결 : 공통 인터랙션 모듈화
버튼에 공통적으로 적용하고 싶은 인터랙션 효과는 크게 두 가지였습니다.

(1) 눌렀을 때 살짝 줄어드는 스케일 효과 (2) 배경색이 변하는 효과 입니다.

 

기존 방식대로라면 각 styled 파일마다 아래와 같은 코드를 반복해서 적어줘야 되었겠죠.

```css
transition: background-color 0.2s ease, transform 0.2s ease;

&:active {
  background-color: #f53e41;
  transform: scale(0.95);
}
``` 

이 문제를 해결하기 위해, 효과를 재사용 가능한 함수 형태로 분리했습니다.
```js
// 버튼 눌림 시 배경색이 변하는 효과
export const backgroundPressEffect = (color: string) => css`
  transition: background-color 0.2s ease;

  &:active {
    background-color: ${color};
  }
`;

// 버튼 눌림 시 살짝 줄어드는 스케일 효과
export const scalePressEffect = (scale: number = 0.95) => css`
  transition: transform 0.2s ease;

  &:active {
    transform: scale(${scale});
  }
`;
 ```

이 방식을 선택한 이유는, 각 효과를 함수로 나누면 조합과 확장이 쉬워지기 때문입니다.
필요한 효과만 선택해서 적용할 수 있어, 버튼마다 다른 인터랙션을 유연하게 구성할 수 있습니다.

 

이제 각 컴포넌트에서는 필요한 효과만 불러와 간단하게 적용할 수 있습니다.
```js
export const Button = styled.button`
  ${backgroundPressEffect('#f53e41')}
  ${scalePressEffect(0.95)}
`;
 ```

또한 이 방식의 장점은 한 번의 수정으로 전체 인터랙션을 쉽게 바꿀 수 있다는 점입니다.

 

예를 들어 스케일 transition 속도를 0.2s → 0.1s 로 줄이고 싶다면 아래와 같이 한줄만 수정하면 됩니다. 한줄의 변경으로 모든 버튼의 눌림 속도가 동일하게 업데이트 됩니다.
```js
export const scalePressEffect = (scale: number = 0.95) => css`
-  transition: transform 0.2s ease;
+  transition: transform 0.1s ease;

  &:active {
    transform: scale(${scale});
  }
`;
 ```

이렇게 스타일 효과를 모듈화하면서 코드 중복 문제는 해결됐지만, 실제 인터랙션 적용 과정에서 또 다른 문제가 드러났습니다.
바로 터치 상태를 단순한 불리언 값으로만 관리하고 있었다는 점입니다.

 

문제2 : 단순 불리언 상태 관리의 한계 ( isTouch = true/false ) 
기존 코드는 터치 여부를 단순히 true 또는 false 상태값으로만 관리하고 있었습니다.
```js
export const Button = () => {
  const [isTouch, setIsTouch] = useState(false);

  return (
    <StyledButton
      $isTouch={isTouch}
      onTouchStart={() => setIsTouch(true)}
      onTouchEnd={() => setIsTouch(false)}
    >
      TOUCH ME
    </StyledButton>
  );
};


const pressBackgroundEffect = (isTouch: boolean, color: string) => css`
  transition: background-color 0.2s ease;
  background-color: ${isTouch ? color : '#fd6c6e'};
`;
```

하지만 이방식의 치명적인 한계가 있었습니다. 

버튼을 눌렀을 때(press)와 손을 뗐을 때(release) 서로 다른 애니메이션을 적용하고 싶었지만, 불리언 값 하나만으로는 이 두 상태를 구체적으로 구분할 수 없었습니다. 터치의 흐름 ( 누름 → 유지 → 해제 )을 표현하기 어려워, 원하는 수준의 인터랙션을 구현하기 힘들었습니다.

 

해결 : TouchState로 세밀한 상태 제어  (TouchState = 'idle' | 'pressing' | 'releasing')
이 문제를 해결하기 위해 터치 상태를 세 단계로 나누어 관리하는 TouchState 구조를 도입했습니다.

idle: 기본 상태 (아무 동작이 없는 상태)
pressing: 버튼을 누른 순간
releasing: 손을 떼는 순간
이렇게 상태를 세분화함으로써 눌렀을 때와 뗐을 때 서로 다른 애니메이션을 적용할 수 있게 되었습니다.

 

터치가 시작될 때는 pressing, 손을 뗄 때는 releasing 상태로 전환하고, 짧은 딜레이후 idle로 복귀하도록 구현했습니다.

이제 각 상태에 따라 애니메이션 효과를 다르게 지정해주면, 사용자가 버튼을 누를 때와 손을 뗄 때 서로 다른 피드백을 제공할 수 있습니다.
```css
  //눌렀을때
  ${({ $touchState }) =>
    $touchState === 'pressing' &&
    css`
      transform: scale(0.9);
      background-color: #e93a3d;
    `}
    
  //떼었을때
  ${({ $touchState }) =>
    $touchState === 'releasing' &&
    css`
      transform: scale(1.05);
      background-color: #ff7a7c;
    `}
 ```

이렇게 touchState를 도입하면서 더 다양한 버튼 인터랙션을 넣을 수 있었지만 또 새로운 문제가 하나 생겼습니다.

 

문제3: 반복되는 상태로직의 복잡성
바로 상태 관리 로직이 각 컴포넌트 안에 반복적으로 존재한다는 점이었습니다. 버튼 컴포넌트마다 touchState와 이벤트 핸들 (handleTouchStart, handleTouchEnd)를 직접 관리하다 보니 코드가 길어지고, 구조가 점점 복잡해졌습니다.

 

그리고 이 로직은 버튼뿐만 아니라 카드, 리스트 아이템 등 터치 가능한 다른 요소에서도 동일하게 사용될 수 있는 공통 패턴이었습니다. 하지만 매번 컴포넌트마다 터치 상태관리 로직을 복사해 넣는 방식은 유지보수에 비효율적이었고, 터치 애니메이션을 수정할 때마다 모든 컴포넌틀를 일일이 수정해야 했습니다. 

 

해결 : 터치 인터랙션 로직을 훅으로 분리하기 
이 문제를 해결하기 위해, 터치 상태 관리와 이벤트 핸들러를 커스텀 훅(useTouchInteraction)으로 분리했습니다.

이 훅은 pressing → releasing → idle의 상태 전환과, 터치가 끝난 뒤 onClick 콜백을 실행하는 로직을 모두 캡슐화합니다.
```js
export const useTouchInteraction = (onClick?: () => void) => {
  const [touchState, setTouchState] = useState<TouchState>('idle');
  const TRANSITION_DELAY_MS = 150;

  const handleTouchStart = useCallback(() => {
    setTouchState('pressing');
  }, []);

  const handleTouchEnd = useCallback(() => {
    setTouchState('releasing');

    // 클릭 콜백 실행
    if (onClick) onClick();

    // 일정 시간 후 idle 상태로 복귀
    setTimeout(() => {
      setTouchState('idle');
    }, TRANSITION_DELAY_MS);
  }, [onClick]);

  return {
    touchState,
    handleTouchStart,
    handleTouchEnd,
  };
};
```

이제 버튼 컴포넌트에서는 훅이 제공하는 상태와 이벤트 핸들러를 가져와서 그대로 넣어주면 됩니다.

원래라면 이 모든 로직이 컴포넌트 내부에 직접 작성되어 있어야 했지만, 훅으로 분리하면서 불필요한 중복이 사라지고 코드 구조가 훨씬 간결해졌습니다.
```js
export const Button = ({ onClick }: Props) => {
  const { touchState, handleTouchStart, handleTouchEnd } =
    useTouchState(onClick);

  return (
    <S.StyledButton
      $touchState={touchState}
      onTouchStart={handleTouchStart}
      onTouchEnd={handleTouchEnd}
    >
      TOUCH ME
    </S.StyledButton>
  );
};
```

최종 구조 정리
위의  과정을 거치며 전체 구조를 다음과 같이 정리할 수 있었습니다.

효과 모듈화 → 배경색 변화와 스케일 변화를 각각 함수로 분리하여 재사용성과 유지보수성을 높임
상태 세분화 → touchState = 'idle' | 'pressing' | 'releasing' 구조를 도입해 세밀한 인터랙션 제어 가능
로직 분리 → 터치 이벤트 핸들링을 useTouchInteraction 훅으로 분리해 코드 길이와 복잡도를 크게 줄임
이렇게 단계적으로 구조를 발전시켜 가면서, 단순히 CSS 효과를 추가하는 것을 넘어 재사용 가능한 인터랙션 시스템으로 발전시킬 수 있었습니다. 이를 통해 버튼뿐만 아니라 다양한 UI 요소에서도 일관된 터치 피드백을 구현할 수 있는 기반이 마련되었습니다.

작은 인터랙션 하나를 구현하는 과정이었지만,코드의 재사용성과 확장성을 함께 고려하면서 작업하니 생각보다 고민할 지점이 많았고, 그만큼 흥미로운 과정이었습니다.


### **적용하고 싶은 효과**

[버튼 인터랙션 설계](https://rundev.tistory.com/19)를 통해 버튼을 눌렀을때, 떼었을때 각각의 효과를 적용할 수 있게 되었습니다.

이제 이를 기반으로 한층 더 세밀한 인터랙션을 구현할 수 있습니다.

제가 적용하고 싶었던 효과는 아래와 같습니다.



눌렀을때 단순이 배경색이 변하는것 뿐만 아니라 **버튼을 누를 때 배경색이 중앙에서 퍼져나갔다가 사라지는 효과**를 구현하고자 했습니다.

이 효과의 핵심은 **::before 가상요소를 활용**하여 가상요소의 scaleX와 opacity를 제어하는 것입니다.

### 

### **가상 요소란 무엇일까?**

가상요소(pseudo-element)는 HTML에 존재하지 않는 **가상의 레이어**를 만들어주는 CSS 기능입니다.

대표적으로 ::before와 ::after 가 있습니다.

가상요소들은 실제 DOM에는 없지만, 렌더링 과정에서 요소 안에 새로운 요소를 만들어주는 것처럼 동작합니다.

이를 이용하면 복잡한 시각적 효과를 HTML 구조를 건드리지 않고도 구현할 수 있습니다.

**그렇다면 왜 ‘HTML 구조를 건드리지 않는 것’이 좋을까요?**

그 이유는 바로 **의미 구조(semantic structure)**를 지킬 수 있기 때문입니다.

**HTML**은 화면의 모양을 그리는 도구가 아니라, **콘텐츠의 논리적 관계와 의미를 표현하는 언어**입니다.

그런데 단순히 시각 효과를 주기 위해 <div>나 <span>을 무분별하게 추가하면, HTML 구조가 불필요하게 복잡해지고 원래의 의미가 흐려집니다.

하지만 가상요소를 사용하면 **HTML 구조는 그대로 유지하면서**, **시각적인 표현만 CSS에서 제어**할 수 있습니다.

### 

### **가상 요소를 사용한 이유**

처음에는 버튼의 실제 배경색을 직접 변경하는 방법도 고려했습니다. 하지만 이 방식대로 구현한다면 버튼 스타일 파일에서 상태에 따른 분기처리가 더 추가되어 코드가 복잡해지고 컴포넌트 구조가 무거워지는 단점이 있었습니다.

반면 ::before를 사용하면, **버튼의 시각적 효과만 별도의 레이어로 분리**할 수 있습니다.

즉, 원본 컴포넌트는 그대로 유지한 채, 그 위에 새로운 레이어를 덧씌우는 방식으로 효과를 표현할 수 있습니다.

이 방식의 장점은 아래와 같습니다.

1. 기존 배경 스타일을 그대로 유지한 상태에서 시각적 효과만 별도로 제어할 수 있으며,

2. 이러한 구조 덕분에 컴포넌트의 재사용성과 유지보수성도 함께 향상됩니다.

### **가상 요소 적용**

이제 가상요소를 사용하여 같이 버튼 효과를 만들어봅시다!

우선 가상요소는 css 에서만 만들 수 있습니다. 아래처럼 &::before 을 쓰면 해당 요소 내부의 가장 앞쪽에 가상 요소가 생깁니다.

```scss
// 버튼 컴포넌트의 컨테이너(Button styled) 안에 가상요소(::before) 추가
&::before {
	content : ""
/* 부모(Button styled)가 position: relative 여야 기준이 잡힘 */
	position:absolute;
	width:100%;
	height:100%;
	top:0;
	left:0;
	background:{$activeColor}
	border-radius:12px
}
```

위 css를 적용한 모습입니다.

원래 버튼 배경색은 분홍색이고 가상요소에는 보라색을 적용해주었습니다.

### **z-index 의 함정**

이 가상요소 레이어는 **버튼 배경색보다는 위**에, **텍스트보다는 아래**에 위치해야합니다. 텍스트보다 위에 위치하면 텍스트까지 가상요소 레이어 색으로 덮이기때문입니다.

::before는 버튼의 **첫 번째 자식요소처럼 렌더링되는 가상 레이어**입니다.

버튼 자식에는 텍스트도 포함되어 있으므로, 이 가상요소가 텍스트보다 뒤쪽에 깔리도록 ::before에 z-index: -1을 적용했습니다.

이러면 가상요소가 버튼의 배경색 위에, 그리고 텍스트 아래에 깔릴 것이라고 예상했습니다.

```
&::before {
	content : ""
	position:absolute;
/*z-index : -1 적용*/z-index:-1;
	width:100%;
	height:100%;
	top:0;
	left:0;
	background:{$activeColor}
	border-radius:12px
}
```

⁉️하지만 제가 예상했던 것과는 다르게 가상요소가 화면에서 아예 사라져버렸습니다


> "z-index는 같은 부모의 자식 요소들끼리 비교되는 게 아니야?"
> 

이 생각이 바로 착각이였습니다.

z-index는 단순히 자식 요소들 간의 순서를 결정하는 속성이 아닙니다.

브라우저는 화면을 여러 개의 **쌓임맥락(stacking context)** 으로 나누고, **각 쌓임 맥락 내부에서 z-index 값을 비교**합니다.

z-index: -1을 적용하더라도 그 요소가 속한 쌓임맥락이 다르면, 위의 예시처럼 의도와는 전혀 다른 위치에 배치될 수 있습니다.

### **쌓임 맥락(stacking context)란?**

쌓임맥락은 브라우저가 요소들을 **화면 위에 어떤 순서로 겹쳐서 그릴지** 결정하기 위해 만드는 **독립된 레이어 공간**입니다.

하나의 페이지 안에는 여러 개의 쌓임맥락이 존재하며, 각 쌓임맥락 안에서는 z-index 값이 큰 요소가 위에, 작은 요소가 아래에 그려집니다.

하지만 다른 쌓임맥락끼리는 서로의 z-index 값을 비교할 수 없습니다**.** 즉, 아무리 z-index: 9999를 줘도, 그 요소가 속한 쌓임맥락 자체가 다른 컨텍스트보다 아래에 있다면 절대 위로 올라갈 수 없습니다.

### **화면에서 무엇이 제일 위에 있는 걸까?**

요소가 화면에서 어떤 순서로 쌓이느냐는 **쌓임맥락(stacking context)** 에 따라 달라집니다.

브라우저는 렌더링할 때 아래 순서로 요소를 쌓습니다.

1. 루트 쌓임맥락
2. 새로운 쌓임맥락을 만드는 요소(예: position + z-index, opacity < 1, transform 등)
3. 각 쌓임맥락 안에서 z-index 값 순으로 정렬

결국 화면에서 가장 위에 보이는 요소는 **가장 높은 쌓임맥락 안에서, z-index 값이 가장 큰 요소**입니다.

### **코드로 보는 stackingContext 작동 예시**

이제 다시 코드로 돌아와, 왜 가상요소가 화면에서 사라졌는지 살펴보겠습니다.

현재 컴포넌트 구조는 아래와 같습니다.

```
<App>
 └── <Button>
      ::before  ← 가상요소
      "Touch me"
```

여기서는 2개의 쌓임 맥락이 존재합니다.

쌓임맥락은 다양한 속성에 의해 생성되지만, 대표적으로 position과 z-index를 함께 선언하면 새로운 쌓임맥락이 자동으로 만들어집니다.

따라서 이 경우는 루트 쌓임맥락과 ::before의 쌓임맥락이 생성됩니다.

```
[Root stacking context] -> 루트 쌓임맥락
 └── Button
      ├── ::before (position: absolute, z-index: -1) → 새로운 쌓임맥락 생성
      └── "Touch me"
```

앞서 설명했듯이 z-index는 **자신이 속한 쌓임맥락 안에서만** 위치가 결정됩니다.

쌓임맥락은 DOM 트리처럼 계층 구조를 가지고 있고, 자신이 속한 쌓임맥락이란 **부모 쌓임맥락**을 의미합니다.

따라서 ::before이 속한 쌓임맥락은 루트 쌓임맥락입니다. (::before 자신의 쌓임맥락이 아님!)

결국 루트 쌓임맥락 내부에는 Button 요소, ::before 가상요소, 그리고 버튼의 텍스트가 모두 포함되어 있습니다.

이 상태에서 ::before에 z-index: -1을 적용하면, 세 요소 중 가장 아래로 내려가며 **버튼보다 뒤로 밀려난 것처럼 보이게 되는 것**이죠.

### **해결 방법**

이 문제를 해결하는 방법은 간단합니다!. 바로 **부모**(**Button)에게 새로운 쌓임맥락을 만들어 주는 것** 입니다.

```
[Root stacking context] → 루트 쌓임맥락
 └── Button → Button 쌓임맥락
      ├── ::before  → before 쌓임맥락
      └── "Touch me"
    
```

이제 ::before이 속한 쌓임맥락은 Button의 쌓임맥락이 됩니다.

Button의 쌓임맥락 내부에는 ::before 가상요소와 텍스트 요소가 존재합니다.

여기서 ::before에 z-index: -1을 적용하면 ::before는 텍스트 아래로 내려갑니다.

::before가 **Button의 쌓임맥락 안에서만 정렬되기 때문에 버튼 자체보다 아래로는 내려가지 않습니다**.

쌓임맥락을 새로 생성하는 방법에는 여러가지가 있지만 여기서는 간단하게 isolation: isolate 를 사용했습니다.

isolation: isolate를 선언하면 해당 요소를 독립된 쌓임맥락으로 분리할 수 있습니다.

```scss
const Button = styled.button`
  position: relative;
	/* 새로운 쌓임맥락 생성 */
	isolation: isolate;

  &::before {
    content: "";
    position: absolute;
    z-index: -1;/* Button 쌓임맥락 내부에서만 뒤로 이동 */width: 100%;
    height: 100%;
    top: 0;
    left: 0;
    background: ${({ $activeColor }) => $activeColor};
    border-radius: 12px;
  }
`;
```



드디어 처음에 의도했던대로 버튼의 배경 위, 텍스트 아래에 ::before 요소가 위치하게 되었습니다!

### **애니메이션 효과 적용**

이제 의도했던 시각적 계층 구조가 완성되었으니, 원하는 효과를 구현할 수 있습니다.

최종적으로 구현한 효과 코드는 아래와 같습니다.

- 상세 코드
    
    ```jsx
    export const pressBackgroundEffect = ({ activeColor, touchState }) => {
      return css`
        position: relative;
        overflow: hidden;
    
        &::before {
          z-index: -1;
          content: '';
          position: absolute;
          top: 0;
          left: 0;
          width: 100%;
          height: 100%;
          background: ${activeColor};
          border-radius: 12px;
    
          transform: scaleX(${getScaleX()});
          opacity: ${getOpacity()};
          transform-origin: center;
          transition: transform ${ANIMATION_DURATION.BACKGROUND_PRESS}ms ease-out,
            opacity ${ANIMATION_DURATION.BACKGROUND_PRESS}ms ease-out;
        }
      `;
    };
    
      const getScaleX = () => {
        switch (touchState) {
          case 'pressing':
            return 1;
          case 'releasing':
            return 1;
          default:
            return 0;
        }
      };
      
      const getOpacity = () => {
        switch (touchState) {
          case 'pressing':
            return 1;
          case 'releasing':
            return 0;
          default:
            return 0;
        }
      };
    ```
    

이 효과는 버튼의 상태 변화에 따라 ::before 가상요소의 scaleX와 opacity 값을 바꾸는 방식으로 작동합니다.

버튼을 누르고 있을 때(pressing)는 배경이 확장되며(scaleX: 1) 완전히 보이도록(opacity: 1) 설정되고,

손을 떼는 순간(releasing)에는 크기는 그대로 유지한 채(scaleX: 1) 투명해지며(opacity: 0) 사라집니다.



### **정리**

이번 문제해결의 핵심은 **가상요소(::before)** 와 **쌓임맥락(stacking context)** 의 원리를 이해해서 의도한 위치에 원하는 요소를 정확히 배치하는 것이었습니다.

처음엔 z-index만 조정하면 될 줄 알았지만, 실제로는 브라우저가 화면에 요소를 어떻게 쌓고 그리는지 알아야 문제를 해결할 수 있었습니다.단순한 효과 하나도 정확히 구현하기 위해서는 배워야 할 것이 많다는 것을 느꼈습니다.
